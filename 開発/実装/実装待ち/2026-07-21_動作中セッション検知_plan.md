# 動作中セッション検知（横断ダッシュボードの3状態化） 実装計画

## 概要

会話ログ方式の質問待ち検知（main済み・`internal/transcript`）を拡張し、各プロジェクトを
**質問待ち(waiting・赤) / 動作中(running・青) / 静観(idle/none)** の3状態で横断表示する。
現状 `parseTail` は「未応答の通常tool_use(busy)」を `IsWaiting=false` で捨てているが、これを
**running として拾い、プロジェクト毎に最新mtimeセッション1つを分類（running/waiting/none）**する。

## 背景・課題

- 現状は「質問待ち」しか出さず、**Claudeが今作業中のセッションが見えない**。
- さらに「作業中の新セッションがあるのに、古い放置セッションの質問待ちが前面に出る」混乱（実j-boardで発生）。
- 当初の「リアルタイム横断把握」に対し、進行中の可視化が欠けている。

## 決定事項（対話＋planner推奨で確定）

| 項目 | 決定 | 根拠 |
|---|---|---|
| 代表セッション | **各プロジェクトの最新mtimeセッションの状態**を表示（古い放置は隠す） | 対話で確定。「作業中なのに古い待機が前面」を解消 |
| 表示 | **3状態を色分けで一望**: 質問待ち赤・最上段 / 動作中青 / 静観灰。並び順 質問待ち>動作中>静観 | 対話で確定 |
| **状態の型設計** | **案C: `idle`(IdleState)は質問待ち専用のまま完全据え置き、`running`(RunningState)を追加フィールドで新設** | 既存「idle存在=質問待ち=赤」契約を壊さず後方互換最大・下流は追加のみ（go-planner推奨） |
| busyThreshold | **45秒**（mtime鮮度でrunningとみなす窓）。45〜60秒窓はrunning継続→60秒(idleMinAge)でwaitingへ一方向遷移 | 応答直後の青→放置で赤、が自然で連続 |
| running の attention | **既存 `progress` を流用**（enum増やさずAPI互換維持・区別は`running`キーで担保） | go-planner推奨 |
| ソート複合キー | 第1=Idle(waiting) > 第2=Running > 第3=既存attention > 経過 > isSelf > name | 質問待ち>動作中>既存 |
| SessionCount | 代表と同一statusのセッション数を維持（代表1件＋件数） | 既存idle集約と同型 |
| 要約 | **動作中は要約しない**（内容が刻々変わる）。summarizerはwaitingのみ。running previewは生text | |
| SSE差分 | `statesDiffer` から `Running.Preview`/`Timestamp` を除外（2秒毎の暴発抑制）。ただしrunningの出現/消滅・running→waiting遷移は必ず検出 | |

## 状態モデル（parse=内容 / reader=mtime鮮度合成）

| 状態 | 判定 | 色/位置 |
|---|---|---|
| **動作中 (running)** | (a)最終実質=mid-turn(未応答通常tool_use / thinking / user末尾でassistant未応答)＝mtime鮮度上限内 / (b)最終=未応答assistantでmtimeAge<60s(idleMinAge) / (c)none-kindでmtimeAge<45s(busyThreshold)。**kind別境界・C-2参照** | 青・中段 |
| **質問待ち (waiting)** | 最終実質=未応答assistant(text/AskUserQuestion) かつ mtimeAge>=idleMinAge(60s) | 赤・最上段 |
| **静観 (none)** | 最新セッションが古い(idleTTL超)or 応答済み・解釈不能 | 灰・下段(バッジ無し) |

- **責務分担**: `parse` は最終エントリ種別(`Kind`: waiting/midTurn/none)まで、**`reader` が mtime 鮮度を合成して最終 status 確定**。
- 遷移: `kind==waiting` かつ mtime<busyThreshold は **running(青)** 扱い、age>=60s で **waiting(赤)** へ。busy誤検知回避（長ツールでmtime止まっても mid-turn で running）。

## 懸念点（planner・決定事項に反映済み）

- **型設計**: 案C（idle据え置き＋running新設）で「idle存在=質問待ち」契約を壊さない。
- **代表選定**: reader が mtime を持つので代表選定は reader 側（entry-time最古選定から mtime最新代表へ）。
- **SSE暴発**: statesDiffer を Running.Preview/Timestamp 除外のフィールド比較に。running出現/消滅・running→waiting遷移は検出必須。
- **要約汚染**: summarizer/summary_cache の対象・aliveKeys を waiting marker のみに（running を haiku 無駄打ちしない）。
- **性能**: parseCache(mtimeキー)流用で mtime不変は再パースskip。
- **thinking末尾**: 生成途中とみなし midTurn(running)。

## レビュー反映（go/nextjs-plan-reviewer・確定・最優先）

実装はこの節を最優先で守る。Critical は方針の誤りでなく記述の不足・不整合。

### バックエンド Critical
- **C-1: attachIdleState は「Status分岐追加」でなく実質"書き換え"**。reader が per-project 代表1件
  （`Marker.Status`＋`Marker.SessionCount`込み）に collapse するので、attachIdleState 側の
  「最古entry-time代表→`SessionCount=len(ms)`」ループ（service.go:206-211）は**削除**する。
  - attachIdleState は MatchProject で ProjectState へ割当 → `rep.Status` で `Idle`/`Running` に
    **ディスパッチのみ** → `SessionCount = rep.SessionCount`（`len(ms)`は捨てる）。
  - **`idleMinAge` ゲート（service.go:189）は waiting marker のみに適用**。running marker はこのゲートを
    通さない（通すと fresh な running が age<60 で落ち**動作中が一切表示されない**）。
- **C-2: しきい値を kind別境界にして 45〜60秒デッドゾーンを解消**。`classifyRepresentative(kind, mtimeAge)`:
  - `midTurn` → running（W-1 の鮮度上限つき）
  - `waiting` → `mtimeAge < 60s(idleMinAge)` は **running**、`>= 60s` は **waiting**（境界は**60sで一本化**・45s使わない）
  - `none` → `mtimeAge < 45s(busyThreshold)` は running、それ以外 none
  - reader が 45s/60s 双方を参照するため**しきい値のSSOTを1箇所**に（busyThreshold と idleMinAge を同一
    パッケージ or transcript が dashboard 定数を import。`idleTTL` が両パッケージ重複している轍を踏まない）。
  - 状態モデル表の「running=mtime<45s」記述は上記に訂正（waitingは60s境界）。

### バックエンド Warning
- **W-1**: midTurn の running を **mtime鮮度上限（例 数分〜十数分）**で切る。上限超の midTurn は none へ
  （Ctrl-C/クラッシュで固まった last=tool_use が最大6h青表示されるのを防ぐ）。しきい値定数化。
- **W-3**: `statesDiffer` は**正規化コピー方式**（Projectsをディープコピーし各 `Running.Preview` を
  ゼロ化して DeepEqual）。手動フィールド列挙は比較漏れを生むため避ける。これで running出現/消滅・
  SessionCount変化・running→waiting遷移は自動検出、preview揮発のみ非broadcast。
- **W-6: `RunningState` から `Timestamp` を落とす（YAGNI）**。RunningBadge は経過分を出さない決定なので
  未使用の churny フィールド。→ **`RunningState{Preview, SessionCount}`** のみ。`Marker.Timestamp` の意味が
  Status で分岐（waiting=entry-time / running=mtime）する点は doc に明記。
- **W-5: running preview のソースを明示**。midTurn 末尾は tool_use/thinking で可読textが無いことが多い→
  「直前の text 要素（`lastTextBefore` 流用）、無ければ空」。空は best-effort（フロントが`(内容なし)`表示）。
- **W-4: `running` 命名の三重定義**（`kanban.running`件数・ops `status:"running"`・`project.running`）。
  → **`project.running` は維持**（両planner一致・状態名と一致・nested/topで区別可）だが、`BACKEND_API.md` と
  型コメントに「3つの running の意味の違い」を明記。
- **W-2**: aliveKeys を **waiting marker のみ**で構築（`summary_cache` 汚染回避）。非代表waitingの要約は
  保持されず代表化時に再生成（初回45s+cooldown遅延）＝**既知挙動として明記**。MergeSummaries は running
  markerにも走るが cacheヒットせず無害（filter不要）。

### フロントエンド Critical
- **FC-1: `isRunningShape` は `timestamp`削除後の `preview:string`/`sessionCount:number` のみ検証、
  `summary`/`summarizedAt` を検証対象に含めない**。含めると全 running が normalizeSnapshot で null化され
  **動作中バッジがサイレントに永久非表示**（エラーも出ない）。TDDに「summaryキー無しrunningが
  `isRunningShape===true`」を必須ケース化。（W-6でTimestampを落としたので検証は preview/sessionCount）

### フロントエンド Warning
- **React.memo は効かない**（state丸ごと再セットで running は毎回新参照）。ちらつき抑制の実体は
  **backend statesDiffer（W-3）**。RunningBadge の memo は外す or プリミティブprops化。「memoで解決」は誤り。
- **normalizeSnapshot は idle/running を1パス・参照維持**（両方正常なら同一 `p` を返す）。片方不正でも
  もう片方/カードを巻き込まない。
- **排他順序を固定**: `const hasRunning = !!project.running && !hasWaiting`。枠色・AccentBar は三項ラダー
  `hasWaiting ? 赤 : hasRunning ? 青 : 既存`（waiting赤pulse > running青static > attention）。
- **preview防御**: RunningBadge は `(running.preview ?? "").trim()`（REST フォールバック経路は normalize を
  通らないため未定義を踏みうる）。空/未定義は `(内容なし)`。
- **`ProjectCardData.running` に「KanbanCounts.running(件数)と別概念」コメント**。

### 変更ファイル/型への反映
- `RunningState{Preview, SessionCount}`（Timestamp削除・W-6）。バック/フロント統一。
- `classifyRepresentative` を kind別境界に。しきい値SSOT 1箇所。midTurn 鮮度上限。
- attachIdleState 書き換え（SessionCount=rep.SessionCount・idleMinAgeゲートwaiting限定・旧選定ループ削除）。
- statesDiffer 正規化コピー方式。isRunningShape summary非検証。

**テストしない判断の根拠:**
- 型/定数の追加（idle.Status・RunningState・Marker フィールド）は挙動を持たず利用側でカバー。
- 実 `~/.claude`／実 SSE／実 `claude` CLI／grasp.sh(bash)／main 配線は結合コスト過大でロジック検証の価値が低い（fake 注入・t.TempDir 固定 JSONL・純粋関数直呼びで代替）。
- readTailBytes/parseCache/allowlist/full-read/壊れ行 skip・useDashboardSSE 再接続は既存テスト済（回帰確認のみ）。
- React.memo は「効かない前提」（W-フロント）でちらつき抑制の実体は backend statesDiffer（W-3）側でテスト。memo の有効性テストは書かない。
- 経過分・色クラス文字列の全網羅や pixel 検証は見た目でコスパ低。クラスの有無 1 点に留める。
