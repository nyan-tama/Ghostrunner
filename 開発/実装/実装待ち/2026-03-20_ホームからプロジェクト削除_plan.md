# ホームからプロジェクト削除 実装計画

## 概要

devtools ホーム画面からプロジェクトを削除できる機能を追加する。削除モードに切り替えるとプロジェクトリスト表示になり、各プロジェクトに削除ボタンが出現。確認後に Docker 停止 + ディレクトリ削除を実行する。

## 確定事項

| 項目 | 決定 |
|------|------|
| 削除対象 | Docker コンテナ・ボリューム停止 + ディレクトリ削除 |
| Git リモート削除 | 含めない |
| パス制限 | ホームディレクトリ直下のみ（`filepath.Dir(cleanPath) == homeDir`） |
| docker compose 失敗時 | ログのみで続行、ディレクトリ削除を実行 |
| docker compose タイムアウト | 30秒 |
| 確認UI | `window.confirm`（MVP） |
| レスポンス形式 | `gin.H{"success": true}` パターン |

## バックエンド計画

### 変更ファイル一覧

| ファイル | 種別 | 内容 | 影響度 |
|---------|------|------|-------|
| `devtools/backend/internal/handler/projects.go` | 修正 | `DestroyRequest` 型と `HandleDestroy` メソッドを追加 | 高 |
| `devtools/backend/cmd/server/main.go` | 修正 | `POST /api/projects/destroy` ルーティング追加 | 低 |

### Step 1: ハンドラーに destroy 機能を追加

**対象**: `devtools/backend/internal/handler/projects.go`

**追加するもの**:
- 型 `DestroyRequest`: フィールド `Path string`（JSON: `path`）
- メソッド `HandleDestroy(c *gin.Context)`: ProjectsHandler のメソッド
- `ProjectsHandler` に `HomeDir string` フィールド追加（テスト時に差し替え可能）

**HandleDestroy の処理フロー**:
1. JSON リクエストのバインド
2. `path` が空でないことを確認 → 400
3. パストラバーサル防止: `filepath.Clean(path)` + `filepath.Dir(cleanPath) == homeDir` でホームディレクトリ直下のみ許可 → 400
4. パスの存在チェック → 404
5. `docker-compose.yml` の存在確認
6. 存在すれば `exec.CommandContext` で `docker compose down -v` を実行（30秒タイムアウト）。失敗してもログ出力のみで続行
7. `os.RemoveAll(cleanPath)` でディレクトリ削除
8. レスポンス `gin.H{"success": true}` or `gin.H{"success": false, "error": "..."}`

### Step 2: ルーティング追加

**対象**: `devtools/backend/cmd/server/main.go`

- `api.POST("/projects/destroy", projectsHandler.HandleDestroy)` を1行追加

### 設計判断

| 判断 | 選択 | 理由 |
|-----|------|------|
| サービス層 | 作らない | ロジックが薄い。既存の ProjectsHandler も直接 OS 操作している |
| docker compose 失敗時 | ログのみで続行 | Docker 未インストール環境でも削除完了させたい |
| compose ファイル名 | `docker-compose.yml` のみ | テンプレートが生成するのはこれのみ |
| パス制限 | ホームディレクトリ直下のみ | `os.RemoveAll` の破壊力を考慮し、`HandleOpen` より厳しい制約 |
| `-v` フラグ | 使用する | ディレクトリごと削除するため。テンプレートのボリュームはプロジェクト固有で共有なし |

## フロントエンド計画

### 変更ファイル一覧

| ファイル | 種別 | 内容 | 影響度 |
|---------|------|------|-------|
| `frontend/src/types/index.ts` | 修正 | `DestroyProjectResponse` 型追加 | 低 |
| `frontend/src/lib/api.ts` | 修正 | `destroyProject` 関数追加 | 低 |
| `frontend/src/components/ProjectDeleteList.tsx` | 新規 | プロジェクトリスト + 削除ボタン | 高 |
| `frontend/src/app/page.tsx` | 修正 | 削除モードトグル、表示切替 | 中 |

### Step 1: 型定義とAPI関数

**対象**: `frontend/src/types/index.ts`, `frontend/src/lib/api.ts`

- 型 `DestroyProjectResponse`: `success: boolean`, `error?: string`
- 関数 `destroyProject(path: string)`: `POST /api/projects/destroy` を呼び出し

### Step 2: ProjectDeleteList コンポーネント

**対象**: `frontend/src/components/ProjectDeleteList.tsx`（新規）

- Props: `projects`, `onDelete`, `deletingPath`
- プロジェクト名・パスのリスト表示
- 各行に赤い「削除」ボタン
- 削除中のプロジェクトはボタン disabled + 「削除中...」表示

### Step 3: page.tsx に削除モードを追加

**対象**: `frontend/src/app/page.tsx`

- State: `isDeleteMode` (boolean), `deletingPath` (string | null)
- 左下に固定配置の「削除モード」トグルボタン
- 削除モード時: CommandForm を非表示、代わりに ProjectDeleteList を表示
- `handleDeleteProject`: `window.confirm` → `destroyProject` → `refreshProjects`
- 削除したプロジェクトが選択中なら `projectPath` を空にリセット
- `refreshProjects` 関数: `fetchProjects` + `setProjects` をまとめる（初回レンダリングでも使用）

### 設計判断

| 判断 | 選択 | 理由 |
|-----|------|------|
| 削除モード時のUI | CommandForm 非表示 + リスト表示 | CommandForm への変更不要でシンプル |
| 確認ダイアログ | `window.confirm` | MVPとして十分 |
| コンポーネント分離 | ProjectDeleteList を新設 | page.tsx（539行）の肥大化防止 |
| エラー表示 | `alert` | MVP |
