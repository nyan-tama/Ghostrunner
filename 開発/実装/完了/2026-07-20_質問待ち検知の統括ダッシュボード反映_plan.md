# 質問待ち検知の統括ダッシュボード反映 実装計画

## 概要

Claude Code のフックで「セッションがユーザーの入力を待って止まっている（質問待ち）」を検知し、
統括ダッシュボードの各プロジェクトに `[質問待ち N分]` ＋「何を待っているかの1行要約」として反映する。
複数プロジェクトを行き来する際の「返事を待たせているプロジェクトの取りこぼし・見失い」を解消する。

検討・実地検証の経緯は [検討ドキュメント](../../検討中/アーカイブ/2026-07-20_統括ダッシュボードのリアルタイム進捗把握.md)
（アーカイブ済）を参照。フック方式は POC ブランチ `feat/idle-marker-poc` で実地検証済み。

## 前提（実地検証済み・確定）

- **検知はフック方式**。`Notification`(`idle_prompt`) が約60秒アイドル後に発火（実機確認済・CLIで発火）、
  `UserPromptSubmit` で解除。フックは stdin JSON で `cwd`(実プロジェクト絶対パス)・`session_id`・
  `transcript_path` を渡す。**cwd がそのまま実パスなので even-terminal/G2 非依存**。
- **マーカー**: `~/.claude/gr-idle-markers/<session_id>.idle` に
  `{ cwd, session_id, timestamp, rawTail{lastAssistant,lastPrompt}, summary, summarizedAt }` を書く。
  フックは `rawTail` まで（LLM は呼ばない）。要約はバックエンドが後付け（Phase 1b）。
- **バックエンド** Go / Clean Architecture（`devtools/backend`）。既存 `GET /api/dashboard/state` が
  `internal/dashboard`（`service.go` ステートレス・`GetState` 都度スキャン）で集約。プロジェクト一覧は
  `patrol_projects.json`（`internal/projects/loader.go`）。
- **フロント** Next.js（`devtools/frontend`）。既存 `useDashboard` は 15 秒ポーリング。`usePatrolSSE` に
  堅牢な再接続あり。`page.tsx` は**現状ソートせず map**（並べ替えはフロント未実装）。
- **CLI grasp**（bash 試作）は `/api/dashboard/state` を読む。

## 確定した設計判断（決定事項）

| 項目 | 決定 | 補足 |
|---|---|---|
| 共有データ契約 | 各プロジェクトに入れ子オブジェクト `idle`（null=質問待ちでない） | バック/フロント同一 JSON。下記「データ契約」参照 |
| 並べ替え | **バックエンド側**で `質問待ち > required > progress > watching`（同点は待機分 降順） | grasp CLI・Web が一貫。フロントは map のみ |
| SSE/ポーリング | SSE 接続時はポーリング停止・切断時のみフォールバック（案1） | `PollingToggle` は「フォールバック自動更新の可否」に意味変更 |

## データ契約（`idle` オブジェクト・バック/フロント共有）

各プロジェクト（`ProjectState` / `ProjectCardData`）に追加。**キーが存在すること自体が「質問待ち」を意味する**
（`*IdleState` + `omitempty` は非待機時にキーごと欠落し `null` は出ない。フロントは undefined/null 両対応）。

```
idle?: {                    // キー欠落 = 質問待ちでない（bool フラグは持たない）
  timestamp:    string   // RFC3339。バッジの「N分」はフロントが now - timestamp で算出
  preview:      string   // 1a: rawTail.lastAssistant 先頭80字 / 1b: summary 未生成時の暫定
  sessionCount: number   // 同プロジェクトの質問待ちセッション数（代表1件＝最長待機＋件数）
  summary:      string   // 1b: 「何を待っているか」の日本語1行要約（生成前は ""）
  summarizedAt: string   // 1b: 要約生成時刻（RFC3339・未生成は ""）
}
```

- Go 側 JSON タグ＝TS 側フィールド名を統一（`timestamp`/`preview`/`sessionCount`/`summary`/`summarizedAt`）。
- **経過分はサーバーに載せない（W1）**: `waitingMinutes` を State に持つと毎ティック差分が出て「SSE 常時再送」か
  「表示凍結」の二択になる。`timestamp` のみ載せ、フロントが毎秒 `now - timestamp` で算出。バックエンドの
  ソートは内部で経過を計算して使う（露出しない）。
- **時刻は RFC3339 に統一（W2）**: 既存 `GeneratedAt`/`UpdatedAt` と揃える。
- **TS 型は `idle?: IdleState | null`（W3/FC3）**: 消費側は `project.idle?.…` の optional chaining。
  「キー欠落 or null = 質問待ちでない」。冗長な `waiting` bool は持たない（存在＝待機）。

## フック（bash・共通基盤）

POC（`feat/idle-marker-poc`）の 2 スクリプトを本実装として整え、`settings.json` に恒久設定する。

- `.claude/hooks/mark-idle.sh`（`Notification` idle_prompt）: stdin から `cwd`/`session_id`/`transcript_path`
  を読み、`transcript_path` の JSONL から最後の assistant テキストと最後のユーザー発言を抽出、
  `~/.claude/gr-idle-markers/<session_id>.idle` に `{cwd, session_id, timestamp, rawTail{lastAssistant,
  lastPrompt}, summary:"", summarizedAt:""}` を書く。**LLM は呼ばない**。抽出失敗時は該当を空文字に
  フォールバック（会話ログ JSONL は公式非サポート・バージョンで変わりうるため）。
- `.claude/hooks/unmark-idle.sh`（`UserPromptSubmit`）: 該当 `session_id` のマーカーを削除。
- `settings.json`: `Notification`(`matcher:"idle_prompt"`)→mark、`UserPromptSubmit`→unmark を追加
  （既存フック群は無傷）。**フックはセッション開始時ロード**のため、恒久設定後は新規/再起動セッションで有効。
- POC の学び: 会話途中に足したフックは当該セッションでは無視される（`UserPromptSubmit` のみ送信毎評価）。

## 残要検証（実装時に確認）

- `-p` 非対話（gr-run 系）は入力待ちでアイドルしないため `idle_prompt` は基本発火しない見込み。
  gr-run の質問待ちは従来どおりファイルの `未回答` 経路でカバー（フック経路と棲み分け）。**両者は
  ダッシュボード上で並置**（`未回答` と `質問待ち` は別概念として表示）。
- 会話ログ JSONL は公式非サポート・バージョンで変わりうる → 抽出失敗時は `preview`/`rawTail` 空で
  フォールバック（盤本体は無事）。
- 複数セッション同時 → `session_id` ファイル名で競合回避（POC で確認済）。
- assistant 最後が `tool_use` のみ（テキスト無し）→ `lastAssistant` 空になるケースの表示。

## レビュー反映（go-plan-reviewer / nextjs-plan-reviewer・確定）

計画をレビューし、Critical 6 件・主要 Warning を以下のとおり修正・確定した。実装はこの節を優先する。

### バックエンド Critical
- **C1: Attention 昇格の実行順序**。`determineAttention` は `ScanProject` 内（scanner.go）で走り、その時点で
  `Idle` はまだ nil。→ 対応: `determineAttention` の先頭に「`Idle != nil` なら `required` を返す」条件を足しつつ、
  **`service.go` の `attachIdleState` が `Idle` を付与した後に `state.Attention = determineAttention(state)` を
  再評価**する。変更ファイルに `scanner.go`（条件追加）＋`service.go`（付与後の再評価）を分離して記載。
  `scanner_test.go` に `Idle` 入りケースを追加。
- **C2: ソートの配置と第1キー**。ソートは scanner.go ではなく **`service.go` の `GetState` 内 `sort.SliceStable`**
  （L87-97）。→ 対応: そこの比較関数を改修。比較キーは
  `(idle存在 desc, attentionPriority asc, 経過時間 desc, isSelf asc, name asc)`。質問待ちは attention に畳まず
  **`Idle` の有無を独立の第1キー**にして「質問待ち > 未回答由来 required」を分離。
- **C3: 要約書き戻しのマーカー復活レース（1b）**。`UserPromptSubmit` フックがマーカーを削除するため、
  要約ジョブの read→書き戻し(rename) の間に削除されると **rename が解消済みマーカーを復活**させる。→ 対応:
  `WriteSummary` は rename 直前に「元マーカーが存在し `session_id`＋`timestamp` が read 時点と一致する」ことを
  確認し、不在/不一致なら**書き戻しを破棄**（compare-and-swap 的ガード）。`saveConfigLocked` の単純 temp+rename
  では外部削除に対応できない点を明記。

### バックエンド Warning
- **W4: 要約 CLI 経路の仕様確定**。`summarize.go` は `claude -p <prompt> --model <確定値> --output-format json` で
  `.result` を 1 行トリム、**専用短タイムアウト（30〜60秒。既存の60分は使わない）**、`cmd.Dir` は home 等の
  非依存ディレクトリ、ツール不要（要約のみ）。`--model` の受理値（`haiku` エイリアス可否/完全モデル名）は
  実装着手時に実機確認（残要検証）。テストは CLI 実行を抽象化してモック。
- **W5: `NewService` シグネチャ変更の波及**。影響ファイルに `service_test.go`（3 呼び出し）を追記。破壊を避けるため
  **`idleReader` は nil 許容**（nil 時は空リーダー扱い）とし既存テストの改修を最小化。
- **W6: dashboard 用 SSE は「新設」**。`writeSSEEvents`/`writePatrolSSEEvents` は型固定で流用不可。`setSSEHeaders`
  のみ共有し、**dashboard 用の書き出し関数を新設**。payload は「イベント」ではなく **`DashboardState` スナップ
  ショット全体**（フロントと契約固定）。バッファは小さめ＋最新優先（coalesce）。
- **W7: `MatchProject` のパス境界**。`filepath.Clean` を両辺に適用し、`cwd == projPath ||
  strings.HasPrefix(cwd, projPath+separator)` でセグメント境界を担保（`/a/b` が `/a/bc` に誤マッチしない）。

### フロントエンド Critical
- **FC1: 接続インジケータの競合**。既存 `ConnectionIndicator` は chat SSE(:3456) を表示中。dashboard SSE(:8888) は
  別接続。→ 対応（MVP）: **dashboard 用の 2 つ目のインジケータをヘッダに独立追加**（chat 用配線は壊さない）。
  合成表示は将来。
- **FC2: `useDashboard` の状態機械**。SSE 接続状態 × ユーザー設定 × visibility が interval を奪い合う。→ 対応:
  **単一の派生値 `shouldPoll = fallbackEnabled && connectionStatus !== "connected" &&
  document.visibilityState === "visible"`** を定義し、**interval を start/stop する `useEffect` を 1 本だけ**に集約。
  `connectionStatus` は ref にも同期して visibilitychange の stale closure を防ぐ（既存 `pollingRef` 踏襲）。
- **FC3: `omitempty`/null**。データ契約の修正どおり `idle?: IdleState | null` ＋ optional chaining。

### フロントエンド Warning
- **fe-W4: 接続状態の 3 値化**。`useDashboardSSE` は `usePatrolSSE` の素な複製にせず、**`useChat` 流の
  3 状態（onerror で `reconnecting`、リトライ上限で `offline`）**を採用してから live/reconnecting/offline に
  マップ（バックオフ中を落とさない）。
- **fe-W5: SSE ペイロードは State スナップショット**。`onmessage` は patrol の `{type,data}` エンベロープではなく
  **生 `DashboardState`** を parse（backend W6 と対の契約）。`onEventRef` パターンを維持し再接続ループを防ぐ。
  共通化するなら `usePatrolSSE` を `path`＋`onEvent<T>` でパラメータ化（低結合方針）。
- **fe-W6: AccentBar は独立軸**。質問待ちを attention=required に畳まず、`idle` の有無を独立フラグとして
