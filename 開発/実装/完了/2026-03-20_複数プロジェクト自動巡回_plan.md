# 複数プロジェクト自動巡回 実装計画

## 概要

devtools に「複数プロジェクト自動巡回」機能を追加する。指定プロジェクトをスキャンし、未処理タスク（`開発/実装/実装待ち/` 内のファイル）があれば `claude -p /coding` を最大5並列で実行する。

## 確定事項

| 項目 | 決定 |
|------|------|
| 対象 | 指定プロジェクトのみ（既存プロジェクト一覧から選択） |
| 並列数 | 5個まで並列、超過分は順次キュー |
| トリガー | 手動ボタン + 定期ポーリング(5分) + plan検知 |
| 承認（ツール許可） | 自動承認（bypassPermissions） |
| 承認（設計判断） | ntfy通知 → ダッシュボードで回答（Question型 + 自由テキスト対応） |
| 承認待ち中 | そのプロジェクトは停止、他は進行 |
| /coding の引数 | `@開発/実装/実装待ち/<ファイル名>` を渡す |
| 登録の永続化 | JSONファイルに保存（再起動後も維持、write-to-temp + rename でアトミック書き込み） |
| 複数実装待ちファイル | 先頭1つを実行。完了後に次回巡回で次のファイルを処理 |

### Step 1: 巡回用の型定義

**ファイル:** `devtools/backend/internal/service/patrol_types.go`（新規）

- 型 `PatrolProject`: フィールド `Path`, `Name`
- 型 `PatrolStatus` (string): 定数 `StatusIdle`, `StatusRunning`, `StatusWaitingApproval`, `StatusQueued`, `StatusCompleted`, `StatusError`
- 型 `ProjectState`: フィールド `Project`, `Status`, `SessionID`, `Questions`(既存Question型を単数で保持), `GitLog`, `PendingTasks`, `Error`, `StartedAt`, `UpdatedAt`
- 型 `PatrolEvent`: SSE配信用
  - フィールド `Type`, `ProjectPath`, `State`, `Message`
  - `Type` の値: `project_started`, `project_question`, `project_completed`, `project_error`, `scan_completed`
  - 個別プロジェクトの細粒度StreamEvent（thinking, tool_use等）は中継しない。状態遷移イベントのみ配信
- 型 `ScanResult`: フィールド `Project`, `GitLog`, `PendingTasks`
- 定数 `MaxParallelSlots = 5`, `PollingInterval = 5 * time.Minute`

### Step 2: 巡回サービスの実装

**ファイル:** `devtools/backend/internal/service/patrol.go`（新規）

- インターフェース `PatrolService`:
  - `RegisterProject(path string) error`
  - `UnregisterProject(path string) error`
  - `ListProjects() []PatrolProject`
  - `ScanProjects() []ScanResult`
  - `StartPatrol() error`
  - `StopPatrol()`
  - `ResumeProject(projectPath, answer string) error`
  - `GetStates() map[string]*ProjectState`
  - `StartPolling()`
  - `StopPolling()`
  - `Subscribe() (<-chan PatrolEvent, func())`
- 構造体 `patrolServiceImpl`:
  - フィールド: `projects`(sync.RWMutex保護のmap), `states`(同), `slots`(buffered channelをセマフォとして使用), `claudeService`, `ntfyService`, `subscribers`, `pollingCancel`, `configPath`(JSONファイルパス), `patrolRunning`(排他制御用フラグ)
- コンストラクタ `NewPatrolService(claudeService, ntfyService, configPath)`
  - 起動時にJSONファイルからプロジェクト一覧を読み込み
  - 登録・解除時にJSONファイルに書き出し（write-to-temp + rename）

**巡回実行ロジック:**
- `ScanProjects`: `git log --oneline -5` をexec.Commandで実行、`開発/実装/実装待ち/` をos.ReadDirで読む
- `StartPatrol`: 実行中フラグをチェック（ポーリング+手動の二重実行を防止）→ スキャン → 未処理タスクのあるプロジェクトをセマフォ付きgoroutineで並列実行
- 各プロジェクトの実行（`startProjectExecution`）:
  - `claude -p "/coding @開発/実装/実装待ち/<先頭ファイル名>"` を `ExecuteCommandStream` で実行
  - StreamEventを監視するgoroutineループで `EventTypeQuestion` を検出したら `StatusWaitingApproval` に変更 + ntfy通知 + PatrolEvent(`project_question`)を配信
  - 完了時に `StatusCompleted` + PatrolEvent(`project_completed`)を配信
  - 同一プロジェクトの二重実行はスキップ（StatusがRunningまたはWaitingApprovalならスキップ）
- `ResumeProject`: `ContinueSessionStream` で再開し、`startProjectExecution` と同じgoroutineループを起動して再度StreamEventを監視

**定期ポーリング:**
- `StartPolling`: goroutine + ticker で5分ごとにスキャン → 未処理があれば自動実行
- `StopPolling`: context.Cancel で停止

**subscriber管理:**
- fan-outパターンで全subscriberにPatrolEventを配信
- select + defaultで送信失敗時はスキップ

## 設計判断

| 判断 | 選択 | 理由 |
|-----|------|------|
| 永続化 | JSONファイル（アトミック書き込み） | 再起動後も巡回対象を維持 |
| 並列制御 | buffered channelセマフォ | Go標準パターン、シンプル |
| SSEイベント型 | PatrolEvent（状態遷移のみ） | 細粒度StreamEventは中継しない |
| SSE書き出し | writePatrolSSEEvents（既存パターン踏襲） | writeCreateSSEEventsと同じ構造 |
| フロントSSE | EventSource | 常時接続型（ダッシュボード監視用途）に最適 |
| フック分離 | usePatrolSSE + usePatrol | SSE接続管理と状態管理を分離 |
| 状態保持 | Record<string, PatrolProjectState> | Mapより既存パターンと一貫 |
| プロジェクト登録 | 既存一覧から選択 | パス手入力よりUXが良い |
| 承認UI | Question型（単数） + 自由テキスト | 選択肢と自由入力の両方に対応 |
| プロジェクト解除 | POST /projects/remove | DELETEのCORS問題を回避 |
| 複数実装待ち | 先頭1つを実行 | 完了後に次回巡回で次のファイル |
| Resume後 | 同じgoroutineループを再起動 | startProjectExecutionと同じ監視ロジック |

### 実装時の課題

#### レビューで指摘・修正した点

- **Critical: StopPatrol の context キャンセル実装** - 当初 `StopPatrol` は `patrolRunning` フラグを `false` にするだけだったが、セマフォ待ちのgoroutineが停止しない問題があった。`context.WithCancel` を導入し、セマフォ取得の `select` で `ctx.Done()` を監視する形に修正
- **Critical: ResumeProject のデータ競合修正** - `ResumeProject` 内で `s.states[cleanPath]` から `status` と `sessionID` を読み取る際、ロック解放後にgoroutine内で参照するとデータ競合が発生する可能性があった。ロック内でローカル変数にコピーする形に修正
- **Warning: prompt 変数のデッドコード削除** - 使用されていない変数を削除
- **Warning: saveConfigLocked のエラー伝搬修正** - `saveConfigLocked` のエラーを握り潰していた箇所を、呼び出し元に `return` で返すように修正

#### 技術的に難しかった点

- **monitorStreamEvents のループ設計**: `EventTypeQuestion` 受信時にループから `return` して待機状態に入り、`ResumeProject` で新しいgoroutineを起動して同じ監視ロジックを再利用する設計。Question後にプロセスが停止される前提のため、ループ終了が正しい動作
- **broadcastState でのコピー**: `s.mu.RLock` 内で `ProjectState` をコピーしてからブロードキャストすることで、subscriber側でのデータ競合を防止

### 残存する懸念点

- **StopPatrol の即座停止**: `StopPatrol` はセマフォ待ちのgoroutineはキャンセルできるが、既に `ExecuteCommandStream` を実行中のプロジェクトは `context.Background()` で起動しているため、Claude CLIプロセス自体は即座に停止しない。現状は次のイベント処理時に状態が更新されるまで動き続ける
- **ResumeProject のセマフォ取得**: `ResumeProject` でセマフォを `s.slots <- struct{}{}` でブロッキング取得しているが、全スロットが埋まっている場合にgoroutineが長時間待機する可能性がある。contextによるタイムアウトは未実装
- **patrol.go のファイルサイズ**: 673行で CLAUDE.md の推奨上限（600行）を超過。コア機能が密結合しているため分割は見送ったが、将来的に巡回実行ロジックとJSON永続化を分離することを検討
- **ntfy通知のエラーハンドリング**: `Notify` の戻り値を確認していない（NtfyServiceインターフェースが `error` を返さない設計のため）

9. POST /api/patrol/resume に {"projectPath": "...", "answer": "yes"} を送信して再開

11. POST /api/patrol/polling/start でポーリングを開始（5分間隔で自動巡回）

### 実装時の課題

#### レビューで指摘・修正した点

- **Critical: APIエンドポイント不一致の修正** - 当初 `removePatrolProject` が `DELETE /api/patrol/projects` を使用していたが、バックエンドが `POST /api/patrol/projects/remove` で実装されていたため修正。同様に `sendPatrolAnswer` のエンドポイントが `/answer` だったが `/resume` に修正
- **Warning: ConnectionIndicator のラベル修正** - `connected` 状態のラベルが「接続中」だったが、「接続済み」に修正。「接続中」は `connecting` 状態のラベルと重複していた
- **Warning: AnswerForm にローディング状態追加** - 当初は `isSubmitting` prop がなく、送信中でもボタン連打が可能だった。`isSubmitting` prop を追加し、ボタン・入力フィールドの `disabled` 制御と「送信中...」テキスト表示を実装
- **Warning: PatrolHeader にローディング状態追加** - 当初は `isLoading` prop がなく、巡回開始/停止の API 呼び出し中でもボタンが有効だった。`isLoading` prop を追加し、`disabled` 制御と「処理中...」テキスト表示を実装
- **Warning: ProjectRegister にエラーハンドリング追加** - `fetchProjects` の Promise チェーンに `.catch()` がなく、ネットワークエラー時にUnhandled Promise Rejectionが発生していた。`.catch()` を追加

#### 技術的に難しかった点

- **usePatrolSSE の onEvent ref パターン**: `onEvent` コールバックを `useRef` で保持し、`useEffect` の依存配列から外すことで、EventSource の再接続を防止。React の一般的なパターンだが、SSE 常時接続との組み合わせでは特に重要
- **usePatrol の楽観的更新**: `handleSendAnswer` で API 呼び出し成功後に即座に `status: "running"` と `question: undefined` に更新する楽観的更新を実装。SSE イベントが後から到着して上書きするため、一時的な不整合は許容

### 残存する懸念点

- **ProjectRegister の console.error**: CLAUDE.md のルールでは本番コードに `console.log` 禁止だが、`console.error` はエラーログ用途として使用中。より厳密にはユーザー向けの error state に変換すべきだが、登録パネル内の補助的なエラーのため現状維持
- **isAnswerSubmitting のグローバル状態**: `page.tsx` で `isAnswerSubmitting` を1つの state で管理しているため、複数プロジェクトが同時に承認待ちの場合、1つに回答すると全てのフォームがローディング状態になる。プロジェクト別の管理に変更する必要がある可能性
- **usePatrol の isRunning 管理**: SSE イベント `project_started` で `isRunning = true` に設定するが、全プロジェクトの巡回が完了したタイミングで `false` に戻すロジックがない。巡回停止は `handleStopPatrol` API 呼び出し時のみ `false` になる
- **EventSource の再接続とデータ整合性**: SSE 切断→再接続の間にイベントを取りこぼす可能性がある。再接続時に REST API で最新状態を再取得するロジックは未実装。現状はページ再訪問時のみ REST で再取得
