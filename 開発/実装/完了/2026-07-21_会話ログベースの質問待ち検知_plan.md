# 会話ログ(transcript)ベースの質問待ち検知 実装計画

## 概要

既存の質問待ち検知（Phase 1a/1b・main済み）は Claude Code のフック（Stop等）が
`~/.claude/gr-idle-markers/*.idle` にマーカーを書き、backend が集約する方式。だが
**VS Code 拡張は `Notification(idle_prompt)` を発火せず、`AskUserQuestion`（選択肢提示の回答待ち）
に対応するフックも飛ばない**ため、待機を取りこぼす（実運用の j-board で判明）。

本計画は、フックに依存せず **backend が各プロジェクトの最新セッションの会話ログ JSONL
（`~/.claude/projects/<project-id>/<session-id>.jsonl`）を直接読み**、「最後の実質エントリが
assistant」かつ「一定時間更新なし（＝ユーザー未返信）」を質問待ちと判定する方式に置き換える。
**環境非依存で AskUserQuestion 含む全待機を捕捉**。

## 背景・なぜ

- フックのイベント種別は環境で挙動が違う（VS Code拡張は Notification非発火・Stopは発火、
  AskUserQuestion対応フックは無い＝Elicitationは MCP専用）。この会話で実地確認済み。
- AskUserQuestion は /plan・/discuss で多用され、最も見失いたくない待機。
- 会話ログ直読みなら環境・イベント種別に依存せず全待機を拾える。

## 最大の設計の勘所（既存資産の流用）

既存 `internal/idle` の `Reader` インターフェース `List(ctx) ([]idle.Marker, error)` が**唯一の
差し替え点**。会話ログを読む新実装 `transcriptReader` を作れば、下流
（`dashboard.attachIdleState` / `idleMinAge=60s` / `idleSummarizeDelay=45s` / idle第1キーソート /
SSE `stream.go` / grasp / フロント `WaitingBadge`）は**無改造で流用**できる。要約の書き戻し先だけが
マーカーファイルから別キャッシュに変わる（`idle.Writer` のキャッシュ実装）。

## 会話ログの事実（確認済み）

- パス `~/.claude/projects/<project-id>/<session-id>.jsonl`。project-id = cwd絶対パスの非英数字を
  `-` 置換（`/Users/user/j-board` → `-Users-user-j-board`）。
- 各行JSON。type=assistant/user/attachment/last-prompt/queue-operation/file-history-snapshot/system 等。
  assistant行 `.message.content[]` に `{type:text|thinking|tool_use}`。AskUserQuestion は assistant の
  `tool_use(name=AskUserQuestion)`。最後のユーザー発言は `last-prompt` 型 `.lastPrompt`。
- **公式非サポート形式（バージョンで変わる）**。抽出は行単位 `fromjson?` で壊れ行 skip・改行保持
  （既存 mark-idle.sh と同ロジック）。抽出失敗時フォールバック必須。

## 決定事項（go-planner 推奨を採用・確認事項の回答）

| 項目 | 決定 | 理由 |
|---|---|---|
| 置換 vs 併存 | **段階置換（案C）**: Phase A/B で transcript を主 Reader に単独投入し実機検証 → Phase C でフック撤去 | 合成の常設複雑さを避け、移行期のみ並行観測 |
| 待機判定 | **最終実質エントリ種別＋busyガード（案B改）**: ノイズ行除外後の最後が assistant（末尾text or 未応答 AskUserQuestion tool_use・通常tool_useで結果未着は busy=非待機）。**age/同一性は entry-time（後述C1）**、mtime は短busyガードのみ | busyは content で判定、AskUserQuestionは未応答tool_useで確実。**Timestamp=mtime は破綻するため C1 参照** |
| セッション発見・帰属 | **実cwd読み取り＋`idle.MatchProject`（C2）**: 各セッションの実cwdをJSONLから読みセグメント境界マッチ。glob/deriveProjectID は走査絞り込み専用（外れたら全走査fallback） | 前方一致globは lossy変換で兄弟衝突/サブディレクトリを誤マッチ（C2参照） |
| セッション選定 | 最新の**entry-time**代表、TTL(idleTTL=6h相当)超過は終了扱い除外、SessionCount=待機中数 | 既存 idle 集約と同型 |
| 要約書き戻し | **独立キャッシュ＋key埋め込みCAS（案A）**: `~/.claude/gr-idle-summaries/<sessionID>_<timestamp>.json`。待機が変われば別keyで旧要約が復活しない | 既存 `Writer.WriteSummary(sessionID,expectedTimestamp,...)` シグネチャ維持・Summarizer無改造 |
| 性能 | **mtimeキャッシュ＋末尾tail読み（案B）**: mtime不変なら再パースskip・末尾のみ読む | 2秒tickでも軽量、fsnotifyはYAGNI |
| preview抽出 | **AskUserQuestion質問文優先（案B）**: 末尾が AskUserQuestion tool_use なら `input.questions[].question` を rawTail.LastAssistant に。無ければ text→空 | 本機能の主目的が質問文表示 |
| フック | **Phase Cで撤去（案A）基本、60s遅延が実用問題ならStop補助残置（案B）に緩和** | まず transcript 単独検証してから判断 |
| 表示遅延 | idleMinAge=60s は現行と同じ（現行も表示は60s閾値）。**遅延は現状比で悪化なし** | 既存の表示閾値を流用 |
| 要約キャッシュ場所 | `~/.claude/gr-idle-summaries/`（home配下・実行時ファイル・gitignore外） | マーカーと同様 |
| フォールバック水準 | 抽出失敗時は**保守的に非待機**（取りこぼす方に倒す・誤検知回避優先） | 誤って質問待ちを乱発しない |

## 実装ステップ（Phase分け）

### Phase A: transcriptReader で待機検出し idle 表示（要約なし）
1. `deriveProjectID`・`discoverSessions`（登録プロジェクトごと project-id 前方一致 glob・mtime取得）。
2. `parseTail`: 末尾tail読み、`noiseEntryTypes` 除外後の最終実質エントリが assistant（末尾text or
   未応答 AskUserQuestion tool_use）か判定、`last-prompt`抽出。壊れ行skip・失敗は `ParseOK=false` で
   保守的（非待機）に。
3. `transcriptReader.List`: プロジェクトごと最新代表を選び、待機候補を `idle.Marker` 化
   （**Timestamp=最後のassistantエントリのentry-time（C1・mtieではない）**、Cwd=**実cwd読み取り（C2）**、
   SessionID=ファイル名、RawTail.LastAssistant=末尾テキスト or 質問文、Summary/SummarizedAt=空。
   **List内でMergeSummaries（C3）**）。
4. `parseCache` で mtime 不変時の再パース抑制。
5. `main.go` 配線を `idle.NewReader(markerDir)` → `transcript.NewReader(...)` に差し替え。
- **完了時**: dashboard/state・SSE・grasp・WaitingBadge が transcript ベースで質問待ち表示
  （AskUserQuestion含む）。要約は空（Phase 1a相当）。

### Phase B: 要約キャッシュ
1. `summaryCacheWriter.WriteSummary`: `~/.claude/gr-idle-summaries/<sessionID>_<timestamp>.json` に
   temp+rename。CASは key の timestamp で担保。
2. `MergeSummaries`（reader/attachIdleState側）で `sessionID+timestamp` からキャッシュを引き反映。
3. `main.go` の Summarizer 配線を `NewSummaryCacheWriter(cacheDir)` に差し替え。
4. TTL超過・孤児キャッシュの掃除（List時 or 軽量prune）。
- **完了時**: 既存 Summarizer（30秒tick・haiku・クールダウン）がそのまま動き要約が表示。

### Phase C: フック一本化（撤去 or 補助残置）
1. transcript方式で全待機を実機確認後、フック（`mark-idle.sh`/unmark/settings.json登録）を撤去、
   または Stop即時反映補助のみ残す。
2. `internal/idle` の marker `fileReader`/marker `Writer` を撤去するか整理。既存マーカーは無害・TTLで失効。
3. doc更新。

## 懸念点（go-planner・推奨は決定事項に反映済み）

上表「決定事項」に集約。特に:
- **busy誤検知**: mtime直近（書き込み中）は idleMinAge フィルタで自然除外。
- **AskUserQuestion確実捕捉**: 末尾が未応答 tool_use(AskUserQuestion) を待機条件に含める。
- **形式変動耐性**: 抽出失敗は保守的に非待機＋preview空。tail読み欠損時は全読みフォールバック。
- **要約復活防止**: キャッシュkeyに待機開始timestampを埋め、待機が変われば別keyで旧要約を出さない。

## レビュー反映（go-plan-reviewer・確定・最優先）

実装はこの節を最優先で守る。Critical 3件は「安定同一性」と「最終書き込み時刻(mtime)」の混同が根因。

### C1（最重要）: `Marker.Timestamp` は mtime ではなく「最後の実質assistantエントリ自身の timestamp」
- **問題**: mtime は待機中でも動く（ノイズ行 `file-history-snapshot` の追記、ユーザーが無関係ファイルを
  編集しただけでも Claude Code が transcript に追記しうる）。mtime を Timestamp にすると:
  (a) 要約キャッシュ key `<sid>_<mtime>` が要約生成中(数十秒)に変わり、**要約が孤児化して二度と表示
  されない**＋`needsSummary` 恒真で**haiku churn** (b) `attachIdleState` の age 判定が mtime基準になり、
  **待機中に別ファイルを編集すると質問待ちカードが消える**(false-negative) (c) `statesDiffer` が
  `Idle.Timestamp` 差分で**2秒毎に無駄broadcast**。
- **修正**: `Marker.Timestamp` = **最後の実質 assistant エントリの JSONL 行 `timestamp` フィールド**
  （待機episodeの安定同一性）。`transcriptTail` に entry-time フィールド（例 `LastAssistantAt`）を追加。
  - mtime は「ミリ秒〜数秒の書き込み途中フリッカ抑制」の短busyガード専用に降格（age/同一性には使わない）。
  - 新しい質問（新assistantエントリ）が出れば entry-time が進む→別keyで旧要約を出さない（本来欲しい
    「待機が変われば復活しない」を**正しく**実現）。
  - entry-time 欠落バージョンの fallback は raw mtime ではなく「last-assistant本文の署名(ハッシュ)＋
    初回検出時刻を parseCache に保持」でキーを安定化（raw mtime fallback は churn 再発）。

### C2: セッション帰属は「実cwd読み取り＋既存 `idle.MatchProject`」
- **問題**: project-id は cwd の非英数字を全部 `-` に潰す**不可逆・lossy**変換。前方一致 glob
  `<projid>*` は兄弟衝突（`app` と `app-backend`）・サブディレクトリ起動を**区別不能**で誤マッチ。
- **修正**: エンコード名で帰属を決めない。**各セッションファイルから実 `cwd`（JSONL各行が持つ）を読み、
  既存 `idle.MatchProject(realCwd, projs)` でセグメント境界マッチ**（`/a/b` vs `/a/bc` を守る既存ロジック
  再利用）。`Marker.Cwd` も実cwd由来にする。`deriveProjectID`/glob は「走査ディレクトリの絞り込み(性能)」
  専用に用途限定し、外れたら `projects/` 全走査 fallback（W6）。

### C3: `MergeSummaries` は `transcriptReader.List` 内部で実行（下流無改造の前提）
- **問題**: `attachIdleState` も `Summarizer` も「reader が Summary込みの Marker を返す」前提。マージを
  attachIdleState 側でやると無改造の看板が嘘になり、Summarizer が空Summaryを見て churn。
- **修正**: `MergeSummaries` は **List 内で必ず実行**し、reader は「Summary込みの完成 Marker」を返す契約。

### Warning 反映
- **W1**: busy/tool_use 判定を content で明文化。「末尾が通常 tool_use(Bash/Edit等)で tool_result 未着＝
  busy＝非待機」「AskUserQuestion は後続に tool_result/user が無いことを確認して待機」。mtime は短busy
  ガードのみ。
- **W2**: `tailReadBytes` 窓に parse可能な実質エントリが1つも無ければ **full-read で再走査**。窓は
  AskUserQuestion の長い `input.questions` を想定し十分大きく（数十KB）。
- **W3**: **Phase A では Summarizer を起動しない（or no-op writer 注入）**。transcript セッションに
  `.idle` が無く WriteSummary(marker) が毎回破棄→その前に haiku 実行済みで永久 churn するため。
  Summarizer は Phase B（キャッシュ writer）で起動。
- **W4**: `parseCache` は「check→unlock→IO→lock→store」で IO をロック外に（並行 List を直列化しない）。
- **W5**: 孤児キャッシュ prune を**必須**化（TTL=idleTTL相当 or List時に現存marker以外を掃除）。
- **W6**: `deriveProjectID` の変換規則は公式非サポートで完全一致保証なし→絞り込み専用＋外れたら全走査 fallback。

### 変更ファイル/型への追記
- `transcriptTail` に `LastAssistantAt`(entry-time) と、fallback用の本文署名を持たせる。
- `transcriptReader.List` 内で `MergeSummaries` を呼ぶ（Summary込みで返す）。
- 待機帰属で `idle.MatchProject` を再利用（transcript が idle を import 済みなので追加依存なし）。

- 対象は **Go バックエンド `devtools/backend`** の新規ロジックのみ。テーブル駆動・`t.TempDir`・フェイク注入。
