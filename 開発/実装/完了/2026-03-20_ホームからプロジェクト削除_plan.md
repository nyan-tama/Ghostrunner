# ホームからプロジェクト削除 実装計画

## 概要

devtools ホーム画面からプロジェクトを削除できる機能を追加する。削除モードに切り替えるとプロジェクトリスト表示になり、各プロジェクトに削除ボタンが出現。確認後に Docker 停止 + ディレクトリ削除を実行する。

## 確定事項

| 項目 | 決定 |
|------|------|
| 削除対象 | Docker コンテナ・ボリューム停止 + ディレクトリ削除 |
| Git リモート削除 | 含めない |
| パス制限 | ホームディレクトリ直下のみ（`filepath.Dir(cleanPath) == homeDir`） |
| docker compose 失敗時 | ログのみで続行、ディレクトリ削除を実行 |
| docker compose タイムアウト | 30秒 |
| 確認UI | `window.confirm`（MVP） |
| レスポンス形式 | `gin.H{"success": true}` パターン |

### Step 1: ハンドラーに destroy 機能を追加

**対象**: `devtools/backend/internal/handler/projects.go`

**追加するもの**:
- 型 `DestroyRequest`: フィールド `Path string`（JSON: `path`）
- メソッド `HandleDestroy(c *gin.Context)`: ProjectsHandler のメソッド
- `ProjectsHandler` に `HomeDir string` フィールド追加（テスト時に差し替え可能）

**HandleDestroy の処理フロー**:
1. JSON リクエストのバインド
2. `path` が空でないことを確認 → 400
3. パストラバーサル防止: `filepath.Clean(path)` + `filepath.Dir(cleanPath) == homeDir` でホームディレクトリ直下のみ許可 → 400
4. パスの存在チェック → 404
5. `docker-compose.yml` の存在確認
6. 存在すれば `exec.CommandContext` で `docker compose down -v` を実行（30秒タイムアウト）。失敗してもログ出力のみで続行
7. `os.RemoveAll(cleanPath)` でディレクトリ削除
8. レスポンス `gin.H{"success": true}` or `gin.H{"success": false, "error": "..."}`

### Step 2: ルーティング追加

**対象**: `devtools/backend/cmd/server/main.go`

- `api.POST("/projects/destroy", projectsHandler.HandleDestroy)` を1行追加

### 設計判断

| 判断 | 選択 | 理由 |
|-----|------|------|
| サービス層 | 作らない | ロジックが薄い。既存の ProjectsHandler も直接 OS 操作している |
| docker compose 失敗時 | ログのみで続行 | Docker 未インストール環境でも削除完了させたい |
| compose ファイル名 | `docker-compose.yml` のみ | テンプレートが生成するのはこれのみ |
| パス制限 | ホームディレクトリ直下のみ | `os.RemoveAll` の破壊力を考慮し、`HandleOpen` より厳しい制約 |
| `-v` フラグ | 使用する | ディレクトリごと削除するため。テンプレートのボリュームはプロジェクト固有で共有なし |

### 残存する懸念点

今後注意が必要な点:

- `docker compose down -v` はテスト環境で実際に Docker が動いていないため、テストではコマンド失敗→ログ出力→削除続行のパスしか通っていない。Docker が正常に動作している環境での統合テストは別途必要
- `os.RemoveAll` は権限エラー等で一部ファイルの削除に失敗した場合、部分的にディレクトリが残る可能性がある（Go 標準ライブラリの仕様）
- `homeDir()` メソッドのデフォルト値が `os.UserHomeDir()` に依存しているため、ホームディレクトリが取得できない環境ではフォールバック値 `/Users/user` が使われる

### 計画からの変更点

実装計画に記載がなかった判断・選択:

- `destroyProject` 関数でエラーレスポンスの JSON パースが失敗するケースをハンドリング。`try/catch` で囲み、パース失敗時はステータスコードを含むデフォルトメッセージ `Failed to destroy project (${status})` を使用する設計とした
- `ProjectDeleteList` にプロジェクトが0件の場合の空状態表示（「登録されているプロジェクトがありません」）を追加。計画書では言及されていなかったが UX として必要と判断
- `window.confirm` のメッセージに「この操作はプロジェクトの登録を解除します。」という補足説明を追加
- `refreshProjects` を `useCallback` + `useEffect` で初回レンダリング時にも使用する構成とした。計画書に記載のあった「初回レンダリングでも使用」方針を `useEffect` の依存配列で実現
- テストケース数が計画より少ない。計画では API テスト3件 + コンポーネントテスト6件 = 合計9件だったが、実装では API テスト3件 + コンポーネントテスト3件 = 合計6件。コンポーネントテストの「confirm いいえで何も起きない」「削除成功後に onDeleted が呼ばれる」は、`onDelete` の呼び出し元（`page.tsx` の `handleDeleteProject`）側の責務として省略された。`window.confirm` の呼び出し自体は `page.tsx` の `handleDeleteProject` 内で行われており、`ProjectDeleteList` コンポーネント自体は confirm を呼ばない設計

### 残存する懸念点

今後注意が必要な点:

- `window.confirm` による確認ダイアログは MVP として採用。将来的にカスタムモーダルへの置き換えが望ましい
