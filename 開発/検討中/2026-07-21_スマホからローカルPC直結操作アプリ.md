## 背景と狙い

外出先でもスマホから、ローカルPC（Mac）上で動く各プロジェクトの Claude Code セッションを**直接触りたい**。閲覧（ダッシュボード）ではなく操作が主眼。

ユーザーが挙げた条件を技術要素に射影すると、ほぼ一意に構成が定まる。

| ユーザーの条件 | 技術的実現 |
|----------------|-----------|
| 全プロジェクトをスマホから操作 | 各プロジェクト = 1 tmux セッション（or ウィンドウ） |
| スマホ操作 = ローカルPC操作 | 同一 tmux セッションに直アタッチ |
| PC とスマホを完全一致 | 同じ tmux セッション（同一 PTY。別実装ではない） |
| 経由させない | Tailscale P2P 直結（WireGuard、データは中継しない） |
| どこからでもアクセス | Tailscale |
| スワイプで切替 | tmux ウィンドウ/セッション = プロジェクト |
| Web ではなくアプリ | ネイティブ SSH/tmux アプリ |
| AI は Claude Code | tmux 内で稼働 |
| 進捗閲覧 | おまけ（Ghostrunner GUI 起動時） |

### なぜ tmux が「完全一致・経由なし」の核か

tmux セッションは複数クライアントから同一セッションに attach できる。PC の VS Code ターミナルとスマホが**同じ tmux セッションに attach すれば、中継も別 UI もなく、文字通り同じ PTY を共有**する。「完全一致」は実装で作るものではなく、同一セッションであることの自然な帰結になる。

## 既存資産との関係（重要）

### even-terminal セッション共有検討（2026-05-26）との対比

過去の検討書「even-terminal の将来位置づけ」は、Mac / iPhone Safari / VSCode を**同一 session で結ぶ**ために、`~/.claude/projects/<cwd>/<SID>.jsonl` の **session ID を共有する（案 ζ）** という**間接方式**を採用した。そこでの主要な悩みは:

- claude バイナリの `--resume <SID>` **同時 attach** で parent uuid chain が分岐するリスク → 「書き込みは 1 プロセスに集約」という運用ルールで回避
- VSCode 拡張に任意 SID resume の UI が無い → 「SID コピー → 統合ターミナルで `claude --resume`」の 3 手順運用

今回の **tmux 直アタッチ**は、同じ「多デバイス一致」問題に対する**別解でありながら、上記の悩みを構造的に解消する**:

| 論点 | 案 ζ（jsonl セッションID共有） | 今回（tmux 直アタッチ） |
|------|------|------|
| 一致の単位 | 会話ログ（jsonl）の再生 | 実行中 PTY そのもの |
| 同時 attach | 分岐リスクあり → 書込 1 プロセス集約が必要 | **tmux が本来的にマルチ attach 対応**（同一プロセスを覗く） |
| VSCode との一致 | SID コピー + resume の手数 | VS Code のターミナルも同じ tmux に attach すれば自動一致 |
| リアルタイム性 | SSE で text_delta を流す | tmux がそのまま画面共有 |

→ **含意**: 「VS Code のターミナルで Claude Code を tmux 内起動する」運用に統一すれば、`Mac の VS Code ⇄ スマホ` は tmux attach 一本で真に一致する。even-terminal 由来の SID 共有の複雑さを、今回の構想は迂回できる可能性が高い。ただし **even-terminal / dashboard / Even G2 は別モダリティ**として引き続き共存（音声・一覧性）。今回のアプリはそれらを置き換えるものではなく、「スマホからの生ターミナル操作」という第 5 経路。

### ntfy 通知 → 回答 の既存ループ

Ghostrunner は確認事項発生時に ntfy でスマホ通知する仕組みを既に持つ。**「通知が来た → スマホでそのプロジェクトの Claude Code を開いて即回答」**という導線が本アプリの最頻ユースケースになる。スワイプ横断は「巡回・様子見」向け、通知ディープリンクは「即応」向け。両輪。

## 論点

構成（tmux 直アタッチ + Tailscale）はほぼ確定。残る主論点は 2 つ。

### 論点1: アプリの実体 — 既存アプリで足りるか / 専用アプリを自作するか
### 論点2: スワイプ横断 UX — tmux ネイティブに乗せるか / アプリ側で被せるか

論点2 は論点1 の選択にほぼ従属する（既存アプリなら tmux/タブに乗る、自作なら pager を被せられる）ので、実質は論点1 が主。

## 選択肢（論点1）

### 案 A: 既存アプリ（Blink Shell / Termius）+ tmux + Tailscale

iOS の成熟した SSH ターミナルアプリを使い、各プロジェクトを**アプリのタブ or tmux ウィンドウ**として扱う。

- 構成:
  - Tailscale で Mac に到達 → `ssh mac -t 'tmux attach -t <project>'`
  - **Blink のタブ = プロジェクト**にすると、Blink のタブ間スワイプがそのまま「プロジェクト横断スワイプ」になる（各タブが別プロジェクトの tmux に attach）
  - あるいは 1 タブ内で tmux ウィンドウ切替（prefix + n/p、ステータスバーのウィンドウ一覧）
  - Blink は **mosh** 対応 → スマホのスリープ/回線切替でも接続維持（tmux のセッション永続と二重で堅牢）
- メリット:
  - **実装ゼロ、今日から動く**。条件 (1)(2)(4)(5)(6) を即満たす。(3) も Blink タブスワイプでほぼ満たせる
  - 保守・セキュリティ（SSH 鍵・mosh・Tailscale ACL）は実績あるアプリに委ねられる
  - 「Claude Code の TUI がスマホ画面で実用になるか」という**最大の未知**を、作らずに検証できる
- デメリット:
  - プロジェクト一覧が Ghostrunner の登録（`patrol_projects.json`）と**自動同期しない**（手動でタブ/接続を用意）
  - 進捗の 3 状態（質問待ち赤/動作中青/静観灰）や ntfy 通知ディープリンクと**統合されない**
  - 「Ghostrunner 専用アプリ」というプロダクト像にはならない
- 工数感: 小（Tailscale 設定 + tmux 命名規約 + Blink セットアップのみ）

### 案 B: Ghostrunner 専用ネイティブアプリを自作（SwiftUI + SwiftTerm + Swift SSH）

iOS ネイティブアプリを作り、SSH クライアント（Citadel 等の Swift SSH ライブラリ）+ ターミナルエミュレータ（SwiftTerm）を内蔵。

- 構成:
  - アプリがプロジェクト一覧を取得（Mac 上で `tmux list-sessions`、or Ghostrunner backend を Tailscale 越しに叩く）
  - **SwiftUI `TabView(.page)` で各プロジェクトを 1 ページ = スワイプ横断が第一級 UX**
  - 各ページが SwiftTerm で該当 tmux セッションに attach
  - 進捗 3 状態をページのバッジ/色に反映（devtools backend の既存 API を流用）。ntfy 通知タップ → 該当プロジェクトのページへディープリンク
- メリット:
  - 条件 (3) スワイプ横断が最も自然。進捗おまけ (6) と通知即応を**アプリに統合**できる
  - Ghostrunner の登録情報・3 状態検知と一体化した「専用リモコン」になる
  - Swift は既に技術スタック内（swift-macos テンプレート、swift-* エージェント一式）
- デメリット:
  - **実装コストが実在**。アプリ内 SSH + 端末エミュレーション（PTY・リサイズ・キーボード拡張バー・エスケープシーケンス）は非自明
  - Claude Code TUI が本当にスマホで実用かを検証する前に作ると、**作ってから「TUI が狭くて使えない」と判明するリスク**
  - 保守（iOS 更新追従・SSH ライブラリ・審査 or 自己署名配布）が継続コスト
- 工数感: 中〜大（MVP でも SwiftTerm + SSH attach + pager の結線が要る）

### 案 C: 段階導入（A で検証 → 必要なら B）＜推奨＞

まず案 A（Blink + Tailscale + tmux）で**今日から実運用**し、以下の「作らないと分からない前提」を潰す:

- Claude Code の TUI はスマホ画面幅で実用に耐えるか（入力ボックス・差分表示・AskUserQuestion）
- tmux マルチ attach 時の**クライアントサイズ競合**（後述リスク）は許容できるか
- Tailscale 到達性・mosh の接続維持は外出先で安定するか
- スワイプ横断は Blink タブで足りるか、専用 pager が本当に欲しいか

これらが検証できてから、**案 A で埋まらない差分（スワイプ横断の質・進捗統合・通知ディープリンク）だけ**を案 B で作る。専用アプリの価値提案が「スワイプ + 統合」に絞られていることが明確なので、価値が確認できるまで作らない。

- メリット: 最速で価値を得る + 大きな未知を作る前に潰す + 専用アプリの投資判断を実データで下せる
- デメリット: 「専用アプリ」の完成が後ろ倒し（ただし A の期間も実用できているので不利益は小さい）
- 工数感: 小（今すぐ）→ 中〜大（検証後・条件付き）

## 推奨（仮）

**案 C（案 A で即運用・検証 → 差分だけ案 B）を推奨。**

理由:
- 条件の大半（直結・完全一致・どこからでも・Claude Code 操作）は **tmux + Tailscale + 良質な既存ターミナルアプリ**で今日満たせる。ここに自作要素は不要。
- 専用アプリ（案 B）が既存アプリを上回る価値は **スワイプ横断の質・進捗3状態の統合・通知ディープリンク**の 3 点に限られる。この 3 点は「あると嬉しい」レベルで、まず無しで回して本当に必要かを見極められる。
- 最大の未知は「**Claude Code の TUI がスマホで実用か**」。これは案 A なら作らず検証でき、もし不満なら案 B の前提（そもそも生ターミナルで良いのか、専用 UI レイヤが要るのか）自体を見直せる。
- 「アプリと思われる」というユーザーの直感は案 B の完成形で回収する。ただし**回収は検証後**。

## 未解決の論点・確認したいこと

1. **スマホ端末は iOS 想定でよいか**（Blink/SwiftTerm は iOS 前提。Android なら Termux + Tailscale + 別アプリ選定になる）。
2. **tmux マルチ attach のサイズ競合をどう扱うか**（下記リスク R1）。「完全一致」を厳密に取るか、PC 側の使い勝手を優先するか。
3. **接続維持は mosh を使うか**（Blink 前提。tmux のセッション永続だけで足りるなら SSH のみでも可）。
4. **プロジェクト = tmux セッション単位か / 1 セッション内のウィンドウ単位か**（スワイプ粒度と命名規約に影響）。

## 技術的リスク・留意点

- **R1: tmux マルチ attach のクライアントサイズ競合。** 同一セッションに Mac（広い）とスマホ（狭い）が同時 attach すると、tmux はデフォルトで最小クライアントに合わせて Mac 側も縮む。対策候補: セッショングループ（`tmux new-session -t`）、`window-size latest`/`aggressive-resize`、または「スマホ attach 中は Mac を detach」運用。「完全一致」を厳密に望むなら縮みは仕様として許容、PC 使い勝手を守るなら分離が要る。**検証フェーズ（案 A）で体感を確認する論点**。
- **R2: Claude Code TUI のスマホ実用性。** 入力ボックス・ストリーミング・差分・AskUserQuestion が縦狭幅で読めるか。横向き/フォント調整で救えるか。**案 A で最優先に検証。**
- **R3: 入力エルゴノミクス。** Esc/矢印/Enter/長文入力。Blink は追加キーバーを持つ。自作時は同等のキーバーを作る必要。
- **R4: セキュリティ。** Tailscale ACL でスマホ→Mac の SSH を限定、SSH 鍵はスマホの Secure Enclave（Blink/Secretive 等）。パスワード認証は無効化。
- **R5: 書き込み衝突（even-terminal 検討との整合）。** tmux 直アタッチは同一 PTY なので jsonl 分岐問題は起きないが、Even G2/even-terminal 経由の別 session と**同じプロジェクトを二重操作**しないという運用整理は引き続き必要。今回のアプリは「tmux 経路」、even-terminal は「HTTP/SSE 経路」で系統が違う点を使い分けガイドに明記。

## MVP（次の一手・案 A ベース）

1. **tmux 命名規約を決める**: プロジェクト = セッション（例: セッション名 = Ghostrunner の project id）。VS Code のターミナルも「Claude Code は tmux 内で起動」に統一（Mac↔スマホ一致の前提）。
2. **Mac 側 Tailscale + SSH を用意**: Tailscale で Mac 到達、SSH 鍵認証、ACL でスマホ限定。
3. **スマホに Blink Shell を導入**: 各プロジェクトを Blink タブ = `ssh mac -t 'tmux attach -t <id>'`（or mosh）。タブスワイプ = プロジェクト横断。
4. **外出先で実運用して R1/R2/R3 を検証**: TUI 実用性・サイズ競合・接続維持・スワイプ体感を記録。
5. **（条件付き・案 B）検証で「専用アプリの価値あり」と判断したら `/plan`**: SwiftUI TabView(.page) + SwiftTerm + Swift SSH で、スワイプ横断・進捗3状態バッジ・ntfy ディープリンクを統合した専用アプリを設計。
