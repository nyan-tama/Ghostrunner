## 背景と狙い

外出先でもスマホから、ローカルPC（Mac）上で動く各プロジェクトの Claude Code セッションを**直接触りたい**。閲覧（ダッシュボード）ではなく操作が主眼。

ユーザーが挙げた条件を技術要素に射影すると、ほぼ一意に構成が定まる。

| ユーザーの条件 | 技術的実現 |
|----------------|-----------|
| 全プロジェクトをスマホから操作 | 各プロジェクト = 1 tmux セッション（or ウィンドウ） |
| スマホ操作 = ローカルPC操作 | 同一 tmux セッションに直アタッチ |
| PC とスマホを完全一致 | 同じ tmux セッション（同一 PTY。別実装ではない） |
| 経由させない | Tailscale P2P 直結（WireGuard、データは中継しない） |
| どこからでもアクセス | Tailscale |
| スワイプで切替 | tmux ウィンドウ/セッション = プロジェクト |
| Web ではなくアプリ | ネイティブ SSH/tmux アプリ |
| AI は Claude Code | tmux 内で稼働 |
| 進捗閲覧 | おまけ（Ghostrunner GUI 起動時） |

### なぜ tmux が「完全一致・経由なし」の核か

tmux セッションは複数クライアントから同一セッションに attach できる。PC の VS Code ターミナルとスマホが**同じ tmux セッションに attach すれば、中継も別 UI もなく、文字通り同じ PTY を共有**する。「完全一致」は実装で作るものではなく、同一セッションであることの自然な帰結になる。

## 既存資産との関係（重要）

### even-terminal セッション共有検討（2026-05-26）との対比

過去の検討書「even-terminal の将来位置づけ」は、Mac / iPhone Safari / VSCode を**同一 session で結ぶ**ために、`~/.claude/projects/<cwd>/<SID>.jsonl` の **session ID を共有する（案 ζ）** という**間接方式**を採用した。そこでの主要な悩みは:

- claude バイナリの `--resume <SID>` **同時 attach** で parent uuid chain が分岐するリスク → 「書き込みは 1 プロセスに集約」という運用ルールで回避
- VSCode 拡張に任意 SID resume の UI が無い → 「SID コピー → 統合ターミナルで `claude --resume`」の 3 手順運用

今回の **tmux 直アタッチ**は、同じ「多デバイス一致」問題に対する**別解でありながら、上記の悩みを構造的に解消する**:

| 論点 | 案 ζ（jsonl セッションID共有） | 今回（tmux 直アタッチ） |
|------|------|------|
| 一致の単位 | 会話ログ（jsonl）の再生 | 実行中 PTY そのもの |
| 同時 attach | 分岐リスクあり → 書込 1 プロセス集約が必要 | **tmux が本来的にマルチ attach 対応**（同一プロセスを覗く） |
| VSCode との一致 | SID コピー + resume の手数 | VS Code のターミナルも同じ tmux に attach すれば自動一致 |
| リアルタイム性 | SSE で text_delta を流す | tmux がそのまま画面共有 |

→ **含意**: 「VS Code のターミナルで Claude Code を tmux 内起動する」運用に統一すれば、`Mac の VS Code ⇄ スマホ` は tmux attach 一本で真に一致する。even-terminal 由来の SID 共有の複雑さを、今回の構想は迂回できる可能性が高い。ただし **even-terminal / dashboard / Even G2 は別モダリティ**として引き続き共存（音声・一覧性）。今回のアプリはそれらを置き換えるものではなく、「スマホからの生ターミナル操作」という第 5 経路。

### ntfy 通知 → 回答 の既存ループ

Ghostrunner は確認事項発生時に ntfy でスマホ通知する仕組みを既に持つ。**「通知が来た → スマホでそのプロジェクトの Claude Code を開いて即回答」**という導線が本アプリの最頻ユースケースになる。スワイプ横断は「巡回・様子見」向け、通知ディープリンクは「即応」向け。両輪。

## 論点

構成（tmux 直アタッチ + Tailscale）はほぼ確定。残る主論点は 2 つ。

### 論点1: アプリの実体 — 既存アプリで足りるか / 専用アプリを自作するか
### 論点2: スワイプ横断 UX — tmux ネイティブに乗せるか / アプリ側で被せるか

論点2 は論点1 の選択にほぼ従属する（既存アプリなら tmux/タブに乗る、自作なら pager を被せられる）ので、実質は論点1 が主。→ **案C（段階導入）で決着済み。案B の発火条件も明文化（後述「案B発火条件」）。**

## 選択肢（論点1）

### 案 A: 既存アプリ（Blink Shell / Termius）+ tmux + Tailscale

iOS の成熟した SSH ターミナルアプリを使い、各プロジェクトを**アプリのタブ or tmux ウィンドウ**として扱う。

- 構成:
  - Tailscale で Mac に到達 → `ssh mac -t 'tmux attach -t <project>'`
  - **Blink のタブ = プロジェクト**にすると、Blink のタブ間スワイプがそのまま「プロジェクト横断スワイプ」になる（各タブが別プロジェクトの tmux に attach）
  - あるいは 1 タブ内で tmux ウィンドウ切替（prefix + n/p、ステータスバーのウィンドウ一覧）
  - Blink は **mosh** 対応 → スマホのスリープ/回線切替でも接続維持（tmux のセッション永続と二重で堅牢）
- メリット:
  - **実装ゼロ、今日から動く**。条件 (1)(2)(4)(5)(6) を即満たす。(3) も Blink タブスワイプでほぼ満たせる
  - 保守・セキュリティ（SSH 鍵・mosh・Tailscale ACL）は実績あるアプリに委ねられる
  - 「Claude Code の TUI がスマホ画面で実用になるか」という**最大の未知**を、作らずに検証できる
- デメリット:
  - プロジェクト一覧が Ghostrunner の登録（`patrol_projects.json`）と**自動同期しない**（手動でタブ/接続を用意）
  - 進捗の 3 状態（質問待ち赤/動作中青/静観灰）や ntfy 通知ディープリンクと**統合されない**
  - 「Ghostrunner 専用アプリ」というプロダクト像にはならない
- 工数感: 小（Tailscale 設定 + tmux 命名規約 + Blink セットアップのみ）

### 案 B: Ghostrunner 専用ネイティブアプリを自作（SwiftUI + SwiftTerm + Swift SSH）

iOS ネイティブアプリを作り、SSH クライアント（Citadel 等の Swift SSH ライブラリ）+ ターミナルエミュレータ（SwiftTerm）を内蔵。

- 構成:
  - アプリがプロジェクト一覧を取得（Mac 上で `tmux list-sessions`、or Ghostrunner backend を Tailscale 越しに叩く）
  - **SwiftUI `TabView(.page)` で各プロジェクトを 1 ページ = スワイプ横断が第一級 UX**
  - 各ページが SwiftTerm で該当 tmux セッションに attach
  - 進捗 3 状態をページのバッジ/色に反映（devtools backend の既存 API を流用）。ntfy 通知タップ → 該当プロジェクトのページへディープリンク
- メリット:
  - 条件 (3) スワイプ横断が最も自然。進捗おまけ (6) と通知即応を**アプリに統合**できる
  - Ghostrunner の登録情報・3 状態検知と一体化した「専用リモコン」になる
  - Swift は既に技術スタック内（swift-macos テンプレート、swift-* エージェント一式）
- デメリット:
  - **実装コストが実在**。アプリ内 SSH + 端末エミュレーション（PTY・リサイズ・キーボード拡張バー・エスケープシーケンス）は非自明
  - Claude Code TUI が本当にスマホで実用かを検証する前に作ると、**作ってから「TUI が狭くて使えない」と判明するリスク**
  - 保守（iOS 更新追従・SSH ライブラリ・審査 or 自己署名配布）が継続コスト
- 工数感: 中〜大（MVP でも SwiftTerm + SSH attach + pager の結線が要る）

### 案 C: 段階導入（A で検証 → 必要なら B）＜推奨＞

まず案 A（Blink + Tailscale + tmux）で**今日から実運用**し、以下の「作らないと分からない前提」を潰す:

- Claude Code の TUI はスマホ画面幅で実用に耐えるか（入力ボックス・差分表示・AskUserQuestion）
- tmux マルチ attach 時の**クライアントサイズ競合**（後述リスク）は許容できるか
- Tailscale 到達性・mosh の接続維持は外出先で安定するか
- スワイプ横断は Blink タブで足りるか、専用 pager が本当に欲しいか

これらが検証できてから、**案 A で埋まらない差分（スワイプ横断の質・進捗統合・通知ディープリンク）だけ**を案 B で作る。専用アプリの価値提案が「スワイプ + 統合」に絞られていることが明確なので、価値が確認できるまで作らない。

- メリット: 最速で価値を得る + 大きな未知を作る前に潰す + 専用アプリの投資判断を実データで下せる
- デメリット: 「専用アプリ」の完成が後ろ倒し（ただし A の期間も実用できているので不利益は小さい）
- 工数感: 小（今すぐ）→ 中〜大（検証後・条件付き）

## 決定（2026-07-22 確定）

**専用アプリを自作する（案B直行・路線2: ゼロから構築）。**

案C（段階導入: 案A検証→案B）を採っていたが、対話で自作の価値が以下の3点に具体化し、検証を待たず着手する判断に至った:

1. **入力UI分離**（最大の差別化）: tmux画面（読み取り）と専用入力エリア（複数行・送信ボタン）を分離。音声入力で長文を入れても送信前に確認・編集可能。汎用ターミナルでは不可能。
2. **スワイプ統合**: patrol_projects.json から自動生成、3状態バッジ、動的フィルタ、ntfy ディープリンク。「プロジェクトを知っている」スワイプ。
3. **SSH自動再接続**: 画面復帰時に自動でSSH再接続→tmux attach。mosh不要でユーザー体験はほぼ同等。

技術構成:
- **SwiftUI**（iOS ネイティブ・縦型 portrait）
- **SwiftTerm**（MIT、ターミナル描画。Blink も内部で使用）
- **Citadel**（MIT、Swift ネイティブ SSH）
- **mosh なし**（SSH自動再接続で代替。mosh iOS クライアントは Blink 以外に存在せず自作は非現実的）
- ライセンス自由（GPLv3縛りなし）

案A環境（tmux命名規約・`window-size latest`・`gr-tmux.sh`）はそのまま Mac 側基盤として流用。

## 確定した要件（2026-07-21）

1. **スマホ = iOS**（Blink/SwiftTerm いずれも iOS 前提で確定）。
2. **iOS 上は縦型（portrait）で操作**。「コンソールだけだから可能」との見立て。→ 縦幅での TUI 実用性は最大の未検証点（R2）、案 A で最優先に体感確認。
3. **サイズ競合（R1）の方針 = 「一致」はセッション（会話・状態）で取り、画面寸法は独立**。Mac は自由にリサイズして操作、スマホは縦幅。`window-size latest`（直近操作クライアントに追従・リフロー）で両立。ほぼ同時操作しない前提なのでハンドオフは自然。
4. **接続維持 = tmux 永続 + mosh**。Mac は caffeinate でスリープ停止できるが、スマホはスリープ/電源OFF が前提 → 再接続は mosh でシームレスに（tmux セッションは Mac 側で生き続ける）。
5. **スワイプセット管理が要件**: スワイプ = 1 プロジェクトずつ切替。加えて (a) **スワイプで巡る順序を定義**、(b) **スワイプ対象に含めるプロジェクトの取捨選択**、が必要。→ Blink では手動（タブの並び順・開くタブの選択）で近似できるが、管理機能としての第一級サポートは専用アプリ（案 B）の明確な価値。

## 決着した論点（2026-07-22）

1. **プロジェクトの粒度 = tmux セッション単位で確定。** プロジェクト = 1 セッション、**セッション名 = patrol_projects.json の project id**（将来 backend の3状態突合に使える伏線）。Blink タブ = セッション = プロジェクト の 1:1。理由は「検討経緯」2026-07-22 行 (1) 参照（サイズ競合の局所化・タブスワイプ 1:1・VS Code ワークスペース対応）。
2. **スワイプセット管理（要件5）の許容ラインと案B発火条件を明文化。** 案A では「順序 = Blink タブ並び順」「取捨選択 = 開くタブの選択」で手動近似。以下の**案B発火条件**のいずれかが"実際の痛み"になった時点で案B を `/plan`、それまで案A のみで検証。

### 案B発火条件（T1-T4）

| # | 発火トリガー | 案A（Blink）で痛む理由 |
|---|---|---|
| T1 | 常用プロジェクトが二桁 / 新規追加が高頻度 | タブの手動メンテ・並べ替えが破綻 |
| T2 | 動的なスワイプセットが欲しい（「緊急3つ」⇄「全部」を切替） | Blink の静的タブでは都度開閉で実質不可 |
| T3 | ntfy 通知タップ → 該当プロジェクト直行（ディープリンク） | Blink は通知から手動でタブ探し |
| T4 | 進捗3状態バッジ（どのタブが質問待ち赤か）を一目で | Blink のタブには出せない |

→ 検証中にこの4つが痛くなければ案B は作らない。痛みが出た項目が、そのまま案B の価値提案（何を統合すべきか）になる。

## 技術的リスク・留意点

- **R1: tmux マルチ attach のクライアントサイズ競合 → 方針確定（2026-07-21）。** 同一セッションに Mac（広い）とスマホ（縦狭い）が同時 attach すると、tmux はデフォルトで最小クライアントに合わせて Mac 側も縮む。ユーザーは Mac 側で自由にリサイズして操作したい。**採用方針: `window-size latest`（直近に操作したクライアントのサイズに追従、内容はリフロー）**。「一致」は同一セッション（Claude Code の会話・状態）で担保し、画面寸法は一致させない。ほぼ同時操作しない（外出中は Mac 未使用）ためハンドオフは自然。代替: `aggressive-resize`、セッショングループ（`tmux new-session -t`）。**案 A で体感確認する。**
- **R2: Claude Code TUI のスマホ実用性。** 入力ボックス・ストリーミング・差分・AskUserQuestion が縦狭幅で読めるか。横向き/フォント調整で救えるか。**案 A で最優先に検証。**
- **R3: 入力エルゴノミクス。** Esc/矢印/Enter/長文入力。Blink は追加キーバーを持つ。自作時は同等のキーバーを作る必要。
- **R4: セキュリティ。** Tailscale ACL でスマホ→Mac の SSH を限定、SSH 鍵はスマホの Secure Enclave（Blink/Secretive 等）。パスワード認証は無効化。
- **R5: 書き込み衝突（even-terminal 検討との整合）。** tmux 直アタッチは同一 PTY なので jsonl 分岐問題は起きないが、Even G2/even-terminal 経由の別 session と**同じプロジェクトを二重操作**しないという運用整理は引き続き必要。今回のアプリは「tmux 経路」、even-terminal は「HTTP/SSE 経路」で系統が違う点を使い分けガイドに明記。

## 自作アプリの実現方式（2026-07-22 決定）

### 路線比較

| 観点 | 路線1: Blink フォーク | 路線2: ゼロから構築 |
|---|---|---|
| SSH/mosh/端末 | 全部入り | SwiftTerm(MIT) + Citadel(MIT) で自前結線 |
| mosh | あり | **なし**（SSH自動再接続で代替） |
| ライセンス | **GPLv3**（公開時ソース公開必須） | **MIT系**（自由） |
| コードベース | 巨大・学習コスト高 | 小・目的特化 |
| カスタム自由度 | Blink の設計に制約される | 完全自由 |

**分岐点は mosh の要否**。mosh の iOS クライアントは Blink 以外に存在せず自作は非現実的。しかし自作アプリなら SSH 自動再接続（画面復帰検知→SSH再接続→tmux attach を自動実行）で mosh の体験を代替可能。→ **路線2を選択**。

### 自作アプリの差別化ポイント（Blink等の汎用ターミナルとの差）

#### 1. 入力UI分離（最大の差別化）

汎用ターミナルでは tmux の1行入力欄に直接タイプするしかない。

- 長文で横スクロール、見づらい
- 音声入力で改行=Enter → **途中送信事故**のリスク
- カーソル修正が矢印キー頼み

自作アプリの解決:

```
+-----------------------------+
|                             |
|   tmux 画面（読み取り専用） |  <- Claude Code の出力を見る
|                             |
+-----------------------------+
| mic                         |
| ここに指示を入力...          |  <- 専用入力エリア（複数行）
|                             |
|                     [送信]  |  <- 確定してから PTY に流し込む
+-----------------------------+
```

- 複数行の入力エリアを分離。書き終わってから送信（途中送信事故なし）
- iOS 標準の音声入力（マイクボタン）で長文を入れても、送信前に確認・編集可能
- 入力確定後、アプリが tmux の PTY にテキストを流し込む

#### 2. スワイプ統合（プロジェクトを「知っている」スワイプ）

| 観点 | Blink タブ | 自作スワイプ |
|---|---|---|
| プロジェクト一覧 | 手動でタブ作成 | **patrol_projects.json から自動生成** |
| 並び順 | 手動で並べ替え | **優先度順を定義可能** |
| 状態バッジ | なし | **質問待ち赤 / 動作中青 / 静観灰** |
| 動的フィルタ | タブを開閉するしかない | **「緊急3つだけ」<->「全部」の切替** |
| 通知→直行 | 通知見て手動でタブ探す | **ntfy タップ→該当プロジェクトに自動ジャンプ** |

操作の「スワイプ」自体の感触は同じ（どちらも iOS ネイティブのページ送り）。差は中身を知っているかどうか。

#### 3. SSH 自動再接続（mosh 不要の接続維持）

| | mosh | SSH手動 | 自作アプリ（SSH自動再接続） |
|---|---|---|---|
| スマホ復帰時 | 自動（UDPで切れてすらいない） | 手動（切断表示→再接続→tmux attach） | **自動**（画面復帰検知→SSH再接続→tmux attach を自動実行） |
| 回線切替 | 自動 | 手動 | **自動**（NetworkMonitor で検知→再接続） |
| シームレス度 | 最高（途切れない） | 低 | 高（一瞬切れるが自動復帰） |

## MVP（自作アプリ）

### 構成

- SwiftUI（iOS、縦型 portrait）
- SwiftTerm（MIT、ターミナル描画）
- Citadel（MIT、Swift ネイティブ SSH）

4. **スワイプ横断**: SwiftUI TabView(.page) でプロジェクト一覧をスワイプ切替。一覧は Mac 上の `tmux list-sessions` から取得

## Ghostrunner iOS への分離（2026-07-22 決定）

自作アプリは Ghostrunner 本体リポジトリに同居させず、独立プロジェクト **`ghostrunner-ios`** として作る。この機能は本質的に二面（Mac側=サーバー／iOS=クライアント）なので、分離はカップリングを断つのではなく、既にあるネットワーク境界にリポジトリ線を合わせるだけ。

### 切断線（何が残り・何が出るか）

| | 残す（Ghostrunner = サーバー側/統括ハブ） | 出す（`ghostrunner-ios` = クライアント） |
|---|---|---|
| Mac側基盤 | `gr-tmux.sh`・tmux命名規約・`~/.tmux.conf`・Makefile(`tmux-ls/up`) | — |
| データ源 | `patrol_projects.json`・devtools backend API（3状態検知） | それを**消費**する側 |
| コード | — | SwiftUI + SwiftTerm + Citadel アプリ一式・Xcodeプロジェクト |
| Git/配布 | 本体の履歴 | 独立リポジトリ・独立バージョン・TestFlight |
| ドキュメント | この検討書（分離決定の原本・Mac側基盤） | アプリの実装計画（`/plan` 出力）は `ghostrunner-ios/開発/` へ |

- 新リポジトリだが `patrol_projects.json` に登録するので、chief-director の横断把握・ntfy・3状態検知はそのまま効く（分離してもハブから見える）。
- API契約（backend 3状態・patrol_projects.json スキーマ）のクロスリポ調整は通常のクライアント/サーバーと同じ軽度な調整で済む。

### iOS エージェント家系（Ghostrunner 本体に新設・汎用）

`ghostrunner-ios` を開発するための6ロールを、go-\*/nextjs-\*/swift-\* と同じ概念で `ios-*` として新設した。**「iOSアプリ全般」の再利用可能な家系**であり、本アプリ固有（tmux/SSH/SwiftTerm/Citadel/PTY/再接続）は焼き込まない。

|---|---|---|
| `ios-planner` | 計画立案 | Read, Grep, Glob, AskUserQuestion |

| `ios-impl` | 実装 | Read, Write, Edit, Bash, Grep, Glob |

- 設計3原則: **汎用**（固有ドメインはプロジェクト側の `/plan`・docs へ）・**低依存**（標準フレームワーク優先、サードパーティ追加は要justify）・**バトンパス必須**（各エージェント単一責務、残りは次の専門へ）。
