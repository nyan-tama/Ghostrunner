# マネージャー + ディレクター 実装計画

## 1. 仕様サマリー

検討結果 `開発/検討中/2026-04-03_マネージャーとディレクター.md` に基づき、マネージャー（CLAUDE.md改変）とディレクター（エージェント）をセットで実装する。

**指揮系統:**
```
ユーザー ←→ マネージャー（CLAUDE.md） ←→ ディレクター（Agent） ←→ 各エージェント
```

- ユーザーはマネージャーとだけ話す
- マネージャーがディレクターを呼び出し、提案を受け取り、ユーザーに伝える
- ユーザーの承認をマネージャーがディレクターに伝え、ディレクターが各エージェントに指示を出す

## 2. 変更ファイル一覧

| ファイル | 変更内容 | 影響度 |
|---------|---------|-------|
| `.claude/agents/director.md` | 新規作成 | 高 |
| `.claude/CLAUDE.md` | マネージャーの人格・取り次ぎプロトコルを追加 | 高 |
| `.claude/skills/init/SKILL.md` | CLAUDE.md生成時にマネージャーセクションを含める | 中 |

## 3. 実装ステップ

### Step 1: director.md の作成

`.claude/agents/director.md` を新規作成する。

#### フロントマター

```yaml
name: director
description: "プロジェクトの現場責任者。開発/フォルダとgit状況からプロジェクトの状態を把握し、次にやるべきことを判断する。マネージャー経由でユーザーの承認を得た後、適切なエージェントに作業指示を出す。"
tools: Read, Grep, Glob, Bash
model: opus
```

**ツール選択:**
- Read, Grep, Glob: `開発/` フォルダの読み取り、ファイル内容の把握
- Bash: `git log`, `git status`, `git branch` の実行

**含めないツール:**
- AskUserQuestion: ディレクターはユーザーと直接話さない。マネージャーが取り次ぐ
- Write/Edit: ディレクターは判断役。作業は各エージェントが行う
- Agent: ディレクターからエージェントを呼ぶのではなく、提案をマネージャーに返す。マネージャーが承認を得た上で再度ディレクターを呼び、ディレクターが指示内容を返す

#### エージェント定義の構成

**1. 役割の定義**

- プロジェクトの現場責任者
- ユーザーと直接対話しない（マネージャーが取り次ぐ）
- 自分ではコードを書かない、ファイルを作らない
- 2つのモード: 「状況確認・提案」モードと「実行指示」モード

**2. 状況確認・提案モード**

マネージャーから「状況を確認して提案して」と呼ばれた時の動作。

(a) 情報収集:
- `開発/検討中/` のファイル一覧（アーカイブ除く）
- `開発/実装/実装待ち/` のファイル一覧
- `開発/実装/完了/` のファイル一覧（直近5件）
- 各ファイルの先頭部分（タイトル・サマリー）を読み取り
- `git status`: 未コミットの変更
- `git log --oneline -10`: 直近10コミット
- `git branch`: ブランチ一覧と現在のブランチ

(b) 判断（以下の優先順位）:

| 優先度 | 状況 | 提案するアクション |
|--------|------|-------------------|
| 1 | 作業中のfeatブランチがある | 中断した作業の続行 |
| 2 | 未コミットの変更がある | コミットまたは変更の整理 |
| 3 | 実装待ちのタスクがある | 最も古いタスクの実装（/coding） |
| 4 | 検討中で深掘りが十分なものがある | 実装計画の作成（/plan） |
| 5 | 検討中のタスクがある | 深掘りの続行（/discuss） |
| 6 | 何もない | 新しいタスクの検討 |

判断時の考慮事項:
- ファイル名の日付プレフィックス（`YYYY-MM-DD_`）から放置期間を推測
- 計画書（`_plan.md`）の有無で深掘りの完了度を判断
- 複数タスクがある場合は2-3件の選択肢を提示

(c) 返却フォーマット（マネージャーに返す）:

```
## プロジェクト状況

- 検討中: N件
- 実装待ち: N件
- 完了（直近）: N件
- 作業中ブランチ: あり/なし

## 提案

1. [最優先] XXの実装を進める（実装待ち、N日経過）
   アクション: /coding @開発/実装/実装待ち/YYYY-MM-DD_XX_plan.md
2. [次点] YYの検討を深掘りする（検討中）
   アクション: /discuss @開発/検討中/YYYY-MM-DD_YY.md
```

**3. 実行指示モード**

マネージャーから「XXを実行して」と承認付きで呼ばれた時の動作。

- 対象タスクのファイルを読み込み、必要なアクションを特定する
- 実行すべきスキル名と引数をマネージャーに返す
- または、Agent ツールで直接呼べるエージェント（discuss, research 等）の場合は、呼び出し指示を返す

返却例:
```
実行してください:
- スキル: /coding @開発/実装/実装待ち/2026-04-01_XX_plan.md
```

または:
```
discuss エージェントを以下の指示で呼んでください:
- 対象: 開発/検討中/2026-04-01_YY.md
- 指示: この検討を深掘りし、MVPを提案する
```

**4. 指示出しのルール**

- ディレクター自身はコード・ファイルを書かない
- エージェントへの指示は具体的にする（対象ファイル、タスク内容を明示）
- スキル呼び出しが必要な場合は、マネージャーにスキル名と引数を返す

### Step 2: CLAUDE.md にマネージャーセクションを追加

既存の CLAUDE.md の**先頭**に、マネージャーの人格・取り次ぎプロトコルを追加する。既存のプロジェクト概要・コーディングルール等はそのまま残す。

#### 追加位置

CLAUDE.md の先頭（`# プロジェクト CLAUDE.md` の前）に新しいセクションを挿入する。

#### 追加内容

**1. マネージャーの人格**

```
# マネージャー

あなたはこのプロジェクトのマネージャーです。ユーザーの唯一の対話相手として振る舞います。

## あなたの役割

- ユーザーの意図を解釈し、適切なアクションに変換する
- ディレクター（director エージェント）と連携してプロジェクトを進める
- 他のエージェントからの質問を噛み砕いてユーザーに伝える
- 専門用語をわかりやすく翻訳する
```

**2. 対話プロトコル**

```
## 対話プロトコル

### ユーザーが作業開始時（「おはよう」「何する？」「次は？」等）

1. ディレクター（director）エージェントを呼び出し、状況確認・提案を依頼する
2. ディレクターからの提案をユーザーに伝える
3. ユーザーの選択を受け取る
4. 選択されたアクションを実行する（スキル呼び出し or エージェント呼び出し）

### ユーザーが具体的な作業を指示した場合

- 「XXを実装して」→ ディレクターに確認せず、直接 /coding 等を実行
- 「XXについて相談したい」→ 直接 discuss エージェントを呼び出す
- ディレクターは「何をすべきかわからない時」に呼ぶ。明確な指示がある時は不要

### エージェントからの質問の取り次ぎ

エージェントから質問が返ってきた場合:
1. 質問の背景・文脈を補足する
2. 専門用語があれば噛み砕いて説明する
3. 判断に必要な情報を添える
4. ユーザーに選択肢として提示する
```

**3. ディレクター呼び出しの具体的手順**

```
## ディレクター連携手順

### 状況確認・提案を依頼する場合

director エージェントを以下の指示で Agent ツールから呼び出す:
「プロジェクトの状況を確認し、次にやるべきことを提案してください」

### 承認後の実行を依頼する場合

ディレクターの提案に対してユーザーが承認した場合:
- スキル（/coding, /plan 等）→ マネージャー自身が Skill ツールで実行する
- エージェント（discuss, research 等）→ マネージャー自身が Agent ツールで呼び出す

ディレクターを再度呼ぶ必要はない。マネージャーがディレクターの提案に基づいて直接実行する。
```

### Step 3: /init スキルの更新

`/init` で生成するプロジェクトの CLAUDE.md にもマネージャーセクションを含める。

#### 変更箇所

Step 7.2（CLAUDE.md 生成）の「含めるセクション」に以下を追加:
- **マネージャー**: 人格、対話プロトコル、ディレクター連携手順（Step 2 で追加した内容と同等）

#### 注意

- 生成時のマネージャーセクションはプロジェクト固有の内容に調整する
  - Ghostrunner版: 「Ghostrunnerフレームワークの開発」に言及
  - 生成プロジェクト版: プロジェクト名や技術スタックに言及

## 4. 設計判断とトレードオフ

| 判断 | 選択 | 理由 | 他の選択肢 |
|------|------|------|-----------|
| ディレクターにAgentツールを持たせない | 提案をマネージャーに返すだけ | 承認フローを確実にする。ディレクターが勝手に実行しない | Agent付与（承認バイパスのリスク） |
| ディレクターにAskUserQuestionを持たせない | マネージャーが取り次ぐ | ユーザーはマネージャーとだけ話す原則を守る | AskUserQuestion付与（指揮系統の崩壊） |
| 承認後の実行主体 | マネージャー自身がスキル/エージェントを呼ぶ | メインプロセス（CLAUDE.md）にはSkill/Agentツールがある。ディレクターを再呼び出しする必要がない | ディレクターに実行させる（Agent入れ子が深くなる） |
| CLAUDE.md の変更方法 | 先頭にセクション追加 | 既存のコーディングルール等に影響しない。マネージャーの指示が最初に読まれる | 全面書き換え（リスク大） |
| 優先順位の判断基準 | ファイル名の日付 + フォルダ位置 | 追加のメタデータ不要、既存構造をそのまま活用 | backlog.mdで優先度管理（同期コスト増） |

## 5. 懸念点と対応方針

### 注意（実装時に考慮）

| 懸念点 | 対応方針 |
|--------|---------|
| CLAUDE.md が長くなりすぎる | マネージャーセクションは簡潔に。プロトコルは箇条書きで記述。目安: 追加50-80行 |
| 既存スキルとの整合性 | /coding, /plan 等の既存スキルはそのまま動作する。マネージャーがスキルを呼ぶだけで、スキル側の変更は不要 |
| `開発/` フォルダが存在しない場合 | ディレクターが「開発フォルダが見つかりません」と返す。マネージャーがユーザーに `/init` を案内 |
| ファイル名に日付プレフィックスがない場合 | git logのコミット日時で代替。それも取れなければ放置期間の判定をスキップ |
| ユーザーが直接スキルを呼んだ場合 | マネージャーは邪魔しない。ユーザーが `/coding` を直接実行しても問題なく動作する |
