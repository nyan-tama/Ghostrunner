# 検討: 統括ダッシュボードのリアルタイム進捗把握（Web）

## 概要

複数プロジェクトを行き来する際に「そのプロジェクトで何をやっていたか・次の一手は何か」を
見失う課題を解消するため、Ghostrunner の統括ダッシュボードで全プロジェクトの進捗を
リアルタイムに横断把握する。**理想形は Web 画面からの進捗確認**。CLI（`/loop`・`watch`）は
今すぐ試せる繋ぎ手段と位置づけ、本命は Web ダッシュボードのリアルタイム化とする。

## 背景・課題

- ユーザーは個人開発で複数プロジェクトを並行して進めており、プロジェクト間を行き来する。
- 行き来のたびに「最後に何を触っていたか」「次にやることは何か」を毎回思い出す必要があり、
  文脈の復元コストが高い。
- 特に「途中で指示待ち（確認事項に答え待ち・ブロック・一時停止）になっているプロジェクト」が
  どれなのか、一覧で分からない。返事を待たせているプロジェクトを取りこぼす。

## 現状把握（既存インフラ・調査結果）

コードを読んで確認した事実。これらを転用・拡張する前提で設計する。

### リアルタイム配信（SSE）
- **統括GUI（`/dashboard`）は現状 15 秒ポーリングのみで SSE 未使用**
  （`devtools/frontend/src/hooks/useDashboard.ts`、`DASHBOARD_POLL_INTERVAL_MS = 15000`）。
  カンバン／運用の変化はリアルタイムに push されない。
- **patrol サブシステムには本物の SSE push 基盤が実装済み**
  （`GET /api/patrol/stream`、`service/patrol.go` の `Subscribe`/`broadcast`、
  共通ヘルパ `internal/handler/sse.go` の `setSSEHeaders`/`writeSSEEvents`）。
- **堅牢な自動再接続フックも実装済み**
  （`devtools/frontend/src/hooks/usePatrolSSE.ts`、exponential backoff・タブ復帰時再接続）。
- → dashboard 状態の SSE 化は、この patrol 基盤とフックを転用すれば現実的。

### 集約ロジック（既存）
- `GET /api/dashboard/state` が横断集約を返す
  （`internal/handler/dashboard.go`、`internal/dashboard/scanner.go`）。
- スキャン内容: 各プロジェクトのカンバン `.md` 件数（実装待ち/実行中/完了/レビュー）、
  運用/状態 `*.json`、未回答確認事項。
- プロジェクト登録は `devtools/backend/patrol_projects.json`（gitignore・ローカル専用、現在6件）。

### 「指示待ち／待機」の検出シグナル（既に揃っている）
| 待機の正体 | シグナル | 現状 |
|---|---|---|
| 確認事項に答え待ち | `*_plan.md` の `**ステータス**: 未回答` | `scanUnanswered` で検出済み |
| 運用がブロック/一時停止 | 運用/状態 `status: blocked / paused` | scanner で検出済み |
| 実行が固まってる疑い（stale） | `status=running` かつ ModTime > 3時間 | scanner/chief-director で検出済み |
| プロジェクトが質問を投げた瞬間 | patrol SSE `project_question` | SSE 基盤にあり |

### git 文脈
- chief-director は `git -C <path> log --oneline -5` と `git status --short` を参照している。
  「直近の動き」「未コミット差分の有無」は取得できる（dashboard 集約には未組込み）。

### 複数プロジェクトへのプロンプト送信（既存）
- 統括GUI のチャットは even-terminal（`POST /api/prompt`、`cwd` でプロジェクト指定、
  provider=claude）を叩き、`/api/events`(SSE) で応答をライブ受信。**現状は 1 プロジェクト固定**。
- `make g2-all` が 5 プロジェクトを並列起動、各プロジェクトが固有ポート（4xxx・
  `.claude/ports.json` 台帳）で even-terminal を持つ（G2 マルチプロジェクト、main 済み）。
- → 「1 画面で N プロジェクト＋プロンプト送信」は、既存の 1 対 1 チャットを複数ポートへ
  向けて並べる合成で実現可能。ゼロからではない。

## 決定事項（ユーザーとの対話で確定）

1. **本命は Web 画面からの進捗確認**。CLI は繋ぎ・補助。
2. **リアルタイムの作り方 = SSE 化（本物 push）**。dashboard 状態を patrol の SSE 基盤を
   転用して push 化する。ポーリング短縮は繋ぎに留める。
3. **「次の一手（前回の続き）」の情報源 = git 自動 ＋ 軽い手動メモ（ハイブリッド）**。
   git のコミット／未コミット差分／ブランチから自動要約し、`続き.md`（仮）に 1 行メモが
   あれば上乗せする。手間ゼロを基本に、書きたい時だけ精度を足す。
4. **「指示待ち／待機」を一級バッジで強調する**。見失い問題の最重要ピース。
5. **表示範囲は 状態 ＋ 文脈（git・次の一手）＋ 指示待ち**。実行ログのライブ表示は今回スコープ外。
6. **ダッシュボードはエディタではなくコックピット**。VS Code / Claude Code と共存し置き換えない。
   手編集はエディタの役目、指示ベースの編集は even-terminal 経由で可能。

### 未確定（次フェーズで判断）
- プロンプト送信（コックピット化）を今回どこまで含めるか。候補:
  - A: 一覧ライブ＋選んだ 1 個にフォーカスして送信（既存チャット UI 流用・現実的）
  - B: N 個同時にプロンプト入力枠（フルコックピット・画面が重い・N セッション負荷）
  - C: 今回は把握のみ、送信は次段
- 変更内容（git diff）をカードに軽く出すか（オプション）。

## 質問待ち（指示待ち）の検知方式（確定・要検証項目あり）

「セッションがユーザーの回答を待って止まっている」状態を盤に `[質問待ち]` として出す。
検知経路は複数あり、対象セッションの動かし方で使い分ける。**本命は Claude Code フック方式**
（even-terminal 起動・G2・cwd 紐付けの制約から解放され、全セッション横断で効くため）。

### 検知経路の全体像
| 経路 | 対象セッション | 前提 | 状態 |
|---|---|---|---|
| **Claude Code フック（本命）** | 対話・gr-run・素の VS Code/ターミナル 全て | settings.json にフック設定 | 下記で成立確認 |
| ファイルの `未回答`（SSOT） | gr-run/coding の背景実装 | plan.md に `**ステータス**: 未回答` | 既存・盤に反映済 |
| even-terminal `awaiting`（代替） | even-terminal 配下の対話 | サーバー起動（G2 or ハブ）、cwd 紐付け | 代替として保持 |

### フック方式の設計（Claude Code 公式仕様＋実機で確認）
- **ON**: `Notification` フックの `idle_prompt` matcher が「Claude が応答を終えて入力待ち」で発火。
  ノンブロッキングでマーカー書き込みに最適。
- **OFF**: `UserPromptSubmit` フックがユーザー送信時に発火 → マーカー削除。
- **プロジェクト特定**: フックは stdin JSON で `cwd`（実プロジェクトパス）・`session_id`・
  `transcript_path` を渡す。**cwd がそのまま実パスなので even-terminal/G2/ポート紐付けは不要**。
- **マーカー内容**（`<markers>/<session_id>.idle` に書く想定）:
  `{ cwd, session_id, timestamp, rawTail, summary, summarizedAt }`
  - `timestamp` → 「何分待たせているか」を算出（放置＝赤・上位）。5秒＝対話中／23分＝見失い。
  - `rawTail` → 要約の材料（生テキスト）。`transcript_path` の JSONL から**最後の assistant テキスト**を
    `jq -s '[.[]|select(.type=="assistant")|.message.content[]?|select(.type=="text")|.text]|last'`
    で、**最後のユーザー発言**を `last-prompt` 型 `.lastPrompt` で抽出（現バージョンで実機確認済。
    `thinking`/`tool_use` ブロックは自動除外）。フックでは**抽出だけ**行い要約はしない（後述）。
  - `summary` → **最後のやりとりを1行に要約したもの**（例:「ユーザー名を一意にするか／必須にするかの
    回答待ち」）。**これが表示の主役**。生の末尾スライスではなく、文章全体を圧縮した「何を待って
    いるか」の1行。
- **要約の作り方（重要）**: 生の最後の文章は長く、頭/末尾スライスでは要点が落ちる。そこで
  **軽量モデル（Haiku 等）で `rawTail` を 1 行要約**する。ただし全 idle で要約するとコスト過大なので:
  - フックは `rawTail` までを書く（LLM は呼ばない・高速・毎ターン安全）。
  - **要約は遅延・キャッシュ**: マーカーが一定時間（例 2〜3 分）残った＝「見失った」ものだけ、
    バックエンド側で**1 回だけ**要約し `summary`/`summarizedAt` に保存。対話中の一過性 idle は要約しない。
  - 再要約は `rawTail` が変化した時のみ。
- **バックエンド**: dashboard scanner がマーカー dir を巡回し、`cwd` を `patrol_projects.json` の
  プロジェクトパスに前方一致で紐付け → `dashboard.State` に `質問待ち`＋経過分＋`summary` を付与。
  併せて未要約かつ滞留マーカーの要約ジョブを起動。
- **表示**（grasp CLI・Web 共通）: `● 要対応 j-board [質問待ち 23分] 「ユーザー名を一意/必須にするかの回答待ち」`。

### 堅牢性の方針
- **中核（どのプロジェクト・何分待ち）は壊れない**: `cwd`/`timestamp` はフック公式フィールド。
- **`summary` プレビューは段階的に劣化**: 会話ログ JSONL は**公式サポート外でバージョンにより
  フォーマット変更あり**（Claude Code の警告）。
  - 要約前 → `(要約中…)` を表示、または `rawTail` 先頭 N 字を暫定表示。
  - 抽出失敗（フォーマット変更等）→ `(プレビューなし)`。盤本体は無事。

### 実地検証の結果（2026-07-20・POC ブランチ `feat/idle-marker-poc` で確認済み）
実際にフックを仕込んで検証し、本命方式が成立することを確認した。
- **`Notification`(`idle_prompt`) は確実に発火する**。起動時ロードのセッションで、約 66 秒アイドル後に発火。
  ペイロード例: `message="Claude is waiting for your input"`、キーに `cwd`/`session_id`/`transcript_path`/
  `notification_type`/`prompt_id` を含む。CLI（Terminal.app）で発火＝IDE 依存の懸念は否定。
- **`idle_prompt` は即時ではなく約 60 秒のアイドル閾値後に発火**。これは「放置＝見失い」の用途に好都合
  （対話中の一過性 idle を自然に除外できる）。
- **`cwd` / `rawTail` 抽出・マーカー ON/OFF は全て緑**（決定的テストで確認。cwd はプロジェクト実パス、
  rawTail は最後の assistant テキスト＋最後のユーザー発言を現バージョンの JSONL から抽出できた）。
- **`UserPromptSubmit` はプロンプト送信毎に確実に発火**（マーカー削除＝OFF が機能）。
- **重要な運用前提**: フックは**セッション開始時にロード**される。会話の途中で settings.json に足した
  `Notification` フックは当該セッションでは無視される（`UserPromptSubmit` は送信毎評価なので途中追加でも発火）。
  → 本実装ではフックを settings.json に恒久設定するため問題なし。既存セッションには再起動後に効く。
- **複数セッション同時**: 別セッションが同じマーカー dir に `<session_id>.idle` を書いても競合なしを確認。

### 残 要検証（`/plan` 内で潰す）
- `-p` 非対話モード（gr-run 系）は入力待ちでアイドルしないため `idle_prompt` は基本発火しない見込み。
  gr-run の質問待ちは従来どおりファイルの `未回答` 経路でカバー（フック経路とは棲み分け）。
- 会話ログ JSONL の `thinking` 有効時のブロック構成、assistant 最後が `tool_use` のみのケース（rawTail 空）。
- **要約**: 使用モデル（Haiku 等）・要約プロンプト・1 行の長さ上限、要約トリガの滞留閾値（何分で要約するか）、
  要約コストと更新頻度のバランス。要約は日本語で「何を待っているか」を主語つき 1 行にする。
- マーカーの掃除（消し忘れ対策）: セッション異常終了で `UserPromptSubmit` が来ずマーカーが残る場合、
  滞留時間や transcript の更新で失効させる TTL 的な仕組み。

## 選択肢とトレードオフ（リアルタイムの実現手段）

| 手段 | 準備 | リアルタイム性 | コスト | 一望性 | 位置づけ |
|---|---|---|---|---|---|
| CLI `/loop` ＋ chief-director | ゼロ・今すぐ | 間隔（分単位） | 毎回トークン消費 | テキスト・低 | 繋ぎ（賢い秘書） |
| CLI ライブ表示（`watch`/TUI） | 小（既存 API 流用） | 秒単位／変化時 | ほぼゼロ | 中 | 繋ぎ（メーター盤） |
| **Web ダッシュボード SSE 化** | 実装が必要 | **本物 push** | 更新は軽い | **高（カード一望・点灯）** | **本命** |

- CLI ライブ表示は把握データが全てファイル/JSON のため LLM 不要で安価・真の秒単位にできるが、
  自然言語での要約はしない。今すぐ試すなら `make backend` ＋
  `watch -n 2 'curl -s localhost:8888/api/dashboard/state | jq ...'` で成立。
- Web ダッシュボードは常時ライブ・一望・指示待ち点灯に最も向く。既存 SSE 基盤の転用で
  ゼロからよりは軽い。

## MVP 提案（段階実装）

### Phase 1: dashboard 状態の SSE 化＋質問待ち強調（本命の最小形）
- **フック（新規）**
  - `Notification`(`idle_prompt`) → `mark-idle.sh`: stdin から `cwd`/`session_id`/`transcript_path`
    を読み、`rawTail`（最後の assistant テキスト＋最後のユーザー発言）を抽出してマーカー
    `<markers>/<session_id>.idle` を書く。**LLM は呼ばない**（高速・毎ターン安全）。
  - `UserPromptSubmit` → `unmark-idle.sh`: `session_id` のマーカーを削除。
  - settings.json への設定は `.claude/` 変更のためブランチを切る（既存方針）。
- **Backend**
  - `GET /api/dashboard/stream` を新設。patrol の SSE 基盤（`setSSEHeaders`/`writeSSEEvents`/
    Subscribe/broadcast）を転用。
  - 更新トリガは MVP では「サーバー内部で短間隔スキャン → 前回と差分があれば broadcast」で開始
    （実装が軽く、クライアントには即 push に見える）。将来 fsnotify に差し替え可能。
  - scanner がマーカー dir を巡回、`cwd`→プロジェクト前方一致で紐付け。`dashboard.State` に
    「質問待ちフラグ＋経過分＋summary」を集約（従来の 未回答／blocked／paused／stale と並置）。
  - **要約ジョブ**: 滞留（例 2〜3 分）した未要約マーカーの `rawTail` を軽量モデルで 1 行要約し、
    `summary`/`summarizedAt` をマーカーに書き戻す（1 回だけ・変化時のみ再要約）。
- **Frontend**
  - `useDashboard` を SSE 対応化（`usePatrolSSE` の再接続ロジックを流用）。ポーリングは
    フォールバックとして残す。
  - `DashboardCard` に「質問待ち（N分）」バッジ＋`summary`（何を待っているかの1行）を追加し、
    待機中プロジェクトを上位・強調表示。要約前は `(要約中…)`、欠落時は `(プレビューなし)`。
- **CLI（grasp）**
  - 試作スクリプトも同じ state を読むので、`[質問待ち]`＋経過分が自動で反映される。

### Phase 2: 「次の一手（前回の続き）」表示
- **Backend**
  - scanner に git 読み取りを追加（直近コミット・未コミット差分件数・現在ブランチ）。
  - `続き.md`（仮・プロジェクト直下 or `開発/` 配下）があれば 1 行メモを読み取り上乗せ。
  - `dashboard.State` の各プロジェクトに「次の一手」フィールドを追加。
- **Frontend**
  - カードに「直近: <コミット要約>（<経過時間>・未コミット<件数>）」「次の一手: <メモ or 推測>」を表示。

### Phase 3（任意・別検討）: コックピット化（プロンプト送信）
- 一覧ライブ＋クリックで 1 プロジェクトにフォーカスして対話（案 A）を軸に検討。
- 接続先は G2 式（プロジェクトごとの even-terminal 固有ポート 4xxx）で分離。
- 「把握（読み取り・自動）／操作（明示指示）」の原則を UI 上でも分ける。
