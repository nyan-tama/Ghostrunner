## 問題の真相

### ユーザーが訴えた3つの不満は、すべて新 `/dashboard` で既に解決済み

| ユーザーの訴え | 古い `/patrol` の現状 | 新 `/dashboard` の現状 |
|----------------|-----------------------|------------------------|
| 進捗確認できてない | バックエンド `/api/patrol/*` が404でデータが取れない | `GET /api/dashboard/state` が 200 OK で全プロジェクト集約取得済み |
| x-likerの運用フォルダが見れない | 運用フォルダ対応なし（開発カンバンのみ想定） | `運用/状態/*.json` を読み、`auto-follow`／`search-like` 2件とも完全に返している |
| ターミナル指示欄が必要 | チャット入力なし | `ChatInput.tsx` が even-terminal API（`/api/prompt` + SSE）に接続済み、TTS／停止／プロジェクト選択も完備 |

つまり真の問題は「新ダッシュボードが完成済みなのにユーザーに届いていない」こと。実装すべきは新機能ではなく **動線整備と古い patrol の廃止**。

### なぜ届いていなかったか（現状の二重構造）

```mermaid
flowchart LR
    HOME[トップ /] -->|「巡回」ボタン| PATROL[/patrol 古い・廃止候補]
    HOME -.->|動線なし URL手打ち要| DASHBOARD[/dashboard 統括GUI MVP・完成済み]

    PATROL -->|API呼び出し| API1[/api/patrol/* 404]
    DASHBOARD -->|API呼び出し| API2[/api/dashboard/state 200]

    subgraph 古い世界
      PATROL
      API1
    end
    subgraph 新しい世界
      DASHBOARD
      API2
    end

    style PATROL fill:#fdd
    style API1 fill:#fdd
    style DASHBOARD fill:#dfd
    style API2 fill:#dfd
```

トップから辿れるのは古い側だけ。新側はURLを直接知っている人しか到達できない。スマホSafariユーザーがブックマーク経由で古い `/patrol` を開き、機能不足に見えたのは当然の帰結。

## 対象ユーザー

- 自分（横断統括する開発者）
- 主な利用環境: スマホSafari（Tailscale 経由 HTTPS で MacBook の devtools にアクセス）
- 副: PC ブラウザ

## 解決する課題

- 古い `/patrol` がユーザーの正面玄関になってしまい、機能不足に見える
- 新 `/dashboard`（統括GUI MVP）が完成しているのに到達手段が無い
- 二重構造そのものが混乱の原因。整理しないと「巡回ダッシュボードが壊れている」という誤解が再発する

## 選択肢の検討

### 案A: 廃止＆一本化（推奨）

- 概要: トップの「巡回」ボタンを `/dashboard` に向け直す。`/patrol` はリダイレクトに変更。古い patrol 関連コード（フロント・バックエンド）は後続で削除
- メリット:
  - 二重構造がなくなりユーザーが迷わない
  - 古い patrol のバックエンドAPIは既に404、残す価値が無い
  - 統括GUI MVP の存在価値が即座にユーザーに届く
- デメリット:
  - 古いブックマーク経由のアクセスは一度リダイレクトを経由する（許容範囲）
- 工数感: 小（MVP部分は10分以下）

### 案B: 古い patrol を改修して運用表示・チャットを追加

- 概要: `/patrol` を残し、欠けている運用表示／チャット入力を後付け
- メリット: URL とブックマークが維持される
- デメリット:
  - 新 `/dashboard` と機能が重複し、二重メンテになる
  - バックエンドAPI `/api/patrol/*` を復活させる必要があり、`PatrolService` 周りも再整備
  - そもそもなぜ新ダッシュボードを作ったかの整合性が崩れる
- 工数感: 大

### 案C: 当面は両方残し、トップに両リンクを並べる

- 概要: トップに「巡回（旧）」「統括（新）」を並列表示
- メリット: 段階的移行できる
- デメリット:
  - ユーザーの混乱がさらに増す（どちらを開けばよいか分からない）
  - 「とりあえず残す」は技術的負債化しやすい
- 工数感: 小

### 採用: 案A

二重構造そのものがユーザー混乱の根本原因であり、案Cはむしろ悪化させる。古い patrol API は既に404で機能していないため、案B のように維持するメリットも無い。

## MVP提案

**推奨案**: 案A 廃止＆一本化

### MVP範囲（このスコープのみ）

1. **トップ `/` の「巡回」ボタンを差し替え**（最優先・10分以下で完了する想定）
   - リンク先: `/patrol` → `/dashboard`
   - ラベル: 「巡回」→ 「統括」または「ダッシュボード」
   - 該当箇所: `devtools/frontend/src/app/page.tsx` 507-513 行
2. **`/patrol` から `/dashboard` への 301/308 リダイレクト**
   - 古いブックマーク救済のため
   - Next.js の `next.config.js` の `redirects()` で実装可能

### 後続（MVPでは扱わない）

- `devtools/frontend/src/app/patrol/` 配下の削除
- `devtools/frontend/src/components/patrol/` 配下の削除
- `devtools/backend/` 側に残っている `PatrolService` 関連コードの整理
- 上記は方針確定後、`/plan` で別タスクとして扱う

### スマホSafariからのアクセス手順（ユーザー向けメモ）

新 dashboard は元々スマホSafariを主想定として作られている（Tailscale 経由HTTPS）。

- アクセスURL例: `http://<Tailscale名>:3333/dashboard`
  - フロントエンドは Makefile の `make frontend` で 3333 番ポート起動（要確認）
  - Tailscale で MacBook に到達できれば、URL末尾を `/dashboard` にするだけで届く
- ホーム画面に追加: Safari の共有メニュー → 「ホーム画面に追加」で疑似アプリ化
- MVP適用後はトップ `/` の「統括」ボタンから1タップで遷移できる

### チャット入力「文章を送りたい」要件への対応

- 新 `/dashboard` の `ChatInput.tsx` は既に even-terminal API (`/api/prompt` + SSE) に接続済み
- テキスト送信・TTS読み上げ・停止・プロジェクト選択も完備
- **要件は既に満たされている**ため新規実装は不要
- 次ステップ: ユーザーに実機（スマホSafari）で動作確認を依頼

## やらないこと（スコープ外）

- ChatInput の機能追加（クイックコマンド、プロジェクト指定ショートカット等）
- ダッシュボードのレイアウト変更
- バックエンドAPIの追加
- 古い patrol コードの即時削除（後続タスクで実施）
