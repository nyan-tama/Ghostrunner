# 検討: 動作中セッション検知（横断ダッシュボードの3状態化）

## 概要

横断ダッシュボードに「**動作中（agent が今まさに走っている）**」状態を加え、各プロジェクトを
`質問待ち（自分待ち）` / `動作中（Claude作業中）` / `静観（静か）` の**3状態**で一望する。
「今どのプロジェクトで agent が cooking 中か／どれが自分待ちか／どれが静かか」がリアルタイムに分かる。

会話ログ（transcript）方式の延長。既存の質問待ち検知（main済み）に「動作中」を足す。

## 背景・課題

- 現状は「質問待ち（自分待ち）」しか検出せず、**Claude が今作業中のセッションは表示されない**。
- さらに**紛らわしい挙動**: プロジェクトに複数セッションがあると、作業中の新セッションがあるのに
  「古い放置セッションの質問待ち」が前面に出る（実 j-board で発生。76分前の古い待機が表示された）。
- 当初の「リアルタイム横断把握」に対し、"進行中の可視化"が欠けている。

## 現状把握（既存の会話ログ方式）

- `internal/transcript` が各プロジェクトの会話ログ JSONL を末尾 tail 読みし、`parseTail` が
  最終実質エントリ（user/assistant の allowlist）を判定。**未応答 assistant（text / AskUserQuestion）
  ＝質問待ち**、**通常 tool_use 結果未着＝busy＝現状は「非待機」扱いで表示されない**。
- reader は**待機セッションのみ** idle.Marker 化。`attachIdleState`/`idleMinAge=60s`/idle第1キーソート/
  SSE/grasp/WaitingBadge が下流。要約は haiku（質問待ちのみ）。
- 課題: busy を捨てている ＋ 代表セッションが「待機セッション」基準なので、作業中の新セッションが埋もれる。

## 決定事項（対話で確定）

| 項目 | 決定 |
|---|---|
| 代表セッション | **各プロジェクトの最新セッション（mtime最新）の状態を表示**。古い放置セッションは隠す（案A）。「作業中なのに古い待機が前面」の混乱を解消し、現在の状態を反映 |
| 見せ方 | **3状態を色分けで一望**（質問待ち=赤・最上段 / 動作中=青 / 静観=灰）。並び順 `質問待ち > 動作中 > 静観`。動作中も「今やってる内容」を抜粋表示 |

## 状態モデル（3状態）

各プロジェクトの**最新セッション**を分類:

| 状態 | 判定（会話ログ） | 意味 | 色/位置 |
|---|---|---|---|
| **動作中 (running)** | mtime が新しい（Claudeが書き込み中）**または** 会話が途中（未応答の通常tool_use / user送信直後でassistant応答前） | Claudeが今走ってる。**自分待ちではない** | 青・中段 |
| **質問待ち (waiting)** | 最終実質＝未応答assistant（text/AskUserQuestion）＋ mtimeが一定時間更新なし | **自分待ち** | 赤・最上段 |
| **静観 (idle)** | 最新セッションが古い（TTL超）or 応答済み・活動なし | 静か | 灰・下段 |

- **境界はおおよそ60秒**（既存 `idleMinAge=60s` と揃える）: 直近更新＝動作中 / 60秒以上放置＋未応答＝質問待ち。
  細かいしきい値（busyThreshold）は /plan で詰める。
- **busy誤検知の回避**: 長いツール実行（Claudeがツール待ち）で mtime が一時的に止まるケースは、
  「最終エントリが未応答の通常tool_use＝mid-turn」で動作中と判定（mtime単独に頼らない）。

## 設計の勘所（/plan 向けメモ・コードは書かない）

- **既存の `idle` を「セッション状態」に一般化**する方向: reader が最新セッションを分類し、
  状態（running/waiting）＋preview＋timestamp を持つ構造を返す。`IdleState` に `status`（waiting/running）
  を足すか、`SessionState` へ拡張するか（後方互換に注意・/planで判断）。
- **代表選定の変更**: 「待機セッションのみ surface」→「**プロジェクト毎に最新セッションを1つ選び分類**」。
  SessionCount の意味は再定義 or 廃止。
- **preview**: 動作中＝最後のassistantテキスト（今やってること）or `実行中: <tool名>`。質問待ち＝現行。
- **要約**: haiku 要約は質問待ちのみ（動作中は内容が刻々変わるので生テキスト表示）。
- **下流流用**: grasp / Web / SSE / ソート は状態フィールドを見て色・並びを出し分け。WaitingBadge を
  3状態対応に拡張 or 動作中バッジ追加。attention: 質問待ち=required、動作中=progress相当（新レベル or 既存流用）。

## MVP 提案（段階）

- **Phase 1（最小）**: バックエンドで最新セッションを3状態分類（動作中/質問待ち/静観）→ `/api/dashboard/state`
  に状態を載せ → **grasp CLI で3状態色分け表示**。動作中は生preview。
- **Phase 2**: Web の DashboardCard を3状態対応（青の動作中バッジ＋今やってる内容）。SSEでライブ。
- （要約は質問待ちのみで現行流用。動作中の要約は当面なし。）

## 残論点（/plan で詰める）

- busyThreshold（動作中と見なす mtime 鮮度・例30〜60秒）と、質問待ち閾値との連続性。
- 「応答直後（Claude終了直後・ユーザーが読んでる）」の扱い: 動作中でも質問待ちでもない短い遷移期をどうするか。
- IdleState 拡張 vs 新型（後方互換・SSE契約）。
- 複数セッション同時に本当に別状態（片方busy片方waiting）の稀ケースは「最新」で割り切る。
- 動作中の preview 内容（最後のtext / 実行中tool / thinking除外）。
