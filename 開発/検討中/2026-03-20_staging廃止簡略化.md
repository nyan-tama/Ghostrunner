# staging 概念の一旦廃止・簡略化

## 日付
2026-03-20

## 背景

`/init` の聞き込みフローで、DB/ストレージ/Redis を選択した場合に staging ブランチ作成・本番デプロイ準備を提示していた。しかし staging の必要性はインフラ依存の有無ではなくデプロイ戦略の問題であり、判断基準がずれている。

一旦 staging の概念を外してフローをシンプルにし、後で改めて検討する。

## 決定事項

### やること

1. `/init`（SKILL.md）から staging ブランチ作成・本番デプロイ準備ステップを削除
2. テンプレートの `deploy.yml` を `main` ブランチのみのトリガーに変更
   - `branches: [main, staging]` → `branches: [main]`
   - staging 環境分岐ロジックの削除（environment、SUFFIX 等）

### やらないこと（残す）

以下のファイルは将来の再検討用にそのまま残す:
- `/stage` コマンド（`.claude/skills/stage/SKILL.md`）
- `/release` コマンド（`.claude/skills/release/SKILL.md`）
- `staging-manager` エージェント
- `release-manager` エージェント

## デプロイフローの変更

### Before
```
feat → staging → main
         ↓        ↓
      staging環境  本番環境
```

### After
```
feat → main
         ↓
      本番環境
```

## 次のステップ

- `/plan` で `/init` と `deploy.yml` の変更を計画
- staging を再導入する場合は、テンプレート選択とは独立した質問として設計する
